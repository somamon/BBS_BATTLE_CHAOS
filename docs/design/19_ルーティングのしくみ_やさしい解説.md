# 19. ルーティングのしくみ（やさしい解説）

このドキュメントは、**いま実際に動いているルーティング**のしくみを、はじめての人でも追えるように説明したものです。
専門用語は最小限にして、出てきた用語は別ファイル [20_専門用語集.md](20_専門用語集.md) で解説しています。わからない言葉が出たらそちらを見てください。

> 設計の意図や細かい仕様は [14](14_自作ルーティング実装設計.md)・[15](15_自作ルーティング実装ガイド.md)・[16_責務マップ.md](16_責務マップ.md) にあります。本書は「**結局どう動くのか**」を一気に理解するための入門ガイドです。

---

## 0. そもそも「ルーティング」とは？

ブラウザで `https://example.com/thread/5` を開いたとき、サーバーは
**「このURLのときは、どの処理を動かせばいい？」** を決めなければいけません。

この「**URL → 動かす処理**」の振り分けを **ルーティング（routing＝道案内）** と呼びます。

たとえると、会社の電話の**代表番号**のようなものです。

```
かかってきた電話（リクエスト）
      ↓
  「ご用件は？」          ← ルーター（交換手）
      ↓
  担当部署につなぐ         ← コントローラー（担当者）
```

- かかってくる電話 = **リクエスト**（ブラウザからのアクセス）
- 交換手 = **ルーター**（URLを見て担当を決める）
- 担当者 = **コントローラー**（実際に処理して返事を作る）

---

## 1. 全体の流れ（まずは地図）

1本のアクセスが、どこを通って返事になるかの全体像です。

```
ブラウザ
  │  GET /thread/5
  ▼
[ nginx ]  ……… Webサーバー。全部のURLを index.php に集める
  │
  ▼
[ public/index.php ] ……… 唯一の入口（フロントコントローラー）
  │  ① セッション開始
  │  ② リクエストを箱に詰める（Request）
  │  ③ ルート定義を読み込む（routes.php）
  ▼
[ Router ] ……… 交換手。URLに合うルートを探す
  │  ④ resolve()： /thread/5 はどのルート？ → 「/thread/{id}」だ！ id=5 を抜き出す
  │  ⑤ dispatch()： ミドルウェアを通してコントローラーを呼ぶ
  ▼
[ ミドルウェア ] ……… 通る前の検問（CSRFチェックなど）※付いているルートだけ
  │
  ▼
[ ThreadController::show() ] ……… 担当者。返事（Response）を作る
  │
  ▼
[ Response ] ……… 返事の箱。send() でブラウザへ送り返す
  │
  ▼
ブラウザに表示
```

ポイントは2つだけ：

- **入口は1つだけ**（`public/index.php`）。どんなURLでも必ずここを通る。
- **交換手（Router）が振り分ける**。担当（Controller）は自分の仕事だけする。

---

## 2. 登場人物（ファイルと役割）

| ファイル | たとえると | 何をする |
|---|---|---|
| `docker/nginx/*.conf` | 建物の正面玄関 | すべてのURLを `index.php` に集める |
| `public/index.php` | 受付（唯一の入口） | 準備をして Router を呼び、最後に返事を送る |
| `routes.php` | 電話帳（路線図） | 「このURL → この担当」を一覧で書いただけ |
| `Router.php` | 交換手 | URLに合うルートを探して、担当につなぐ |
| `Route.php` | 路線図の1行 | 1本のルートの情報。URLを正規表現に変換する |
| `RouteMatch.php` | 案内メモ | 「見つかったルート＋抜き出した値」を運ぶ入れ物 |
| `Request.php` | 届いた手紙 | アクセス内容（URL・メソッド・入力）を読みやすくした箱 |
| `Response.php` | 返信の手紙 | 返事（HTMLやJSON）を入れて送り返す箱 |
| `Middleware/` | 検問所 | 担当に渡す前のチェック（CSRFなど） |
| `Controller/` | 各部署の担当者 | 実際に処理して Response を作る |

---

## 3. ルートはどう書く？（routes.php）

ルート（道）の定義は [routes.php](../../src/Presentation/Routing/routes.php) に**一覧で**書いてあります。
ここにはロジック（複雑な処理）を書きません。「**どのURLが、どの担当か**」を並べるだけです。

```php
// 静的ルート（URLが固定）
$router->get('/playground', [PlaygroundController::class, 'index']);
$router->get('/thread/create', [ThreadController::class, 'createForm']);

// 動的ルート（URLの一部が変わる。{id} の部分を抜き出す）
$router->get('/thread/{id}', [ThreadController::class, 'show']);
$router->get('/thread/{id:\d+}/posts', [ThreadController::class, 'posts']); // {id:\d+} は「数字だけ」

// ミドルウェア付き（第3引数）
$router->post('/api/thread', [ThreadController::class, 'create'], ['csrf']);
```

読み方はとてもシンプルです。

```
$router->get( '/thread/{id}' , [ThreadController::class, 'show'] );
          │         │                    │              │
       メソッド     URL              コントローラー    メソッド名
       (GET)                          (担当クラス)    (担当の処理)
```

- `get()` … GETリクエスト（ページを見るとき）用
- `post()` … POSTリクエスト（フォーム送信など）用
- `[クラス, 'メソッド名']` … 「**このクラスのこのメソッドを呼んでね**」という指定（呼び出し先）

### 2種類のルート

| 種類 | 例 | 特徴 |
|---|---|---|
| **静的ルート** | `/thread/create` | URLが完全固定。最速で見つかる |
| **動的ルート** | `/thread/{id}` | `{id}` の部分が可変。`/thread/5` でも `/thread/99` でも一致し、`id` を抜き出せる |

`{id}` のように `{ }` で囲んだ部分を **パスパラメータ**（URLの中の変数）と呼びます。
`{id:\d+}` のように `:` のあとに条件を書くと、「**数字のときだけ一致**」のように制限できます（`\d+` は「1文字以上の数字」の意味。詳しくは用語集へ）。

## 4. URLが処理にたどり着くまで（実例で追う）

`GET /thread/5` を例に、1ステップずつ追います。

### ステップ① 玄関：nginx が index.php に集める

[docker/nginx の設定](../../docker/nginx) にこう書いてあります。

```nginx
location / {
    try_files $uri $uri/ /index.php?$query_string;
}
```

意味は「**そのファイルが実在しなければ、ぜんぶ `index.php` に回す**」。
だから `/thread/5` のような“ファイルとして存在しないURL”もすべて `index.php` に届きます。これが「入口が1つ」のしくみです。

### ステップ② 受付：index.php が準備する

[public/index.php](../../public/index.php) がやることは4つだけ。

```php
session_start();                          // ① セッション開始
$request = Request::fromGlobals();        // ② アクセス内容を箱に詰める
$router = new Router();
require '.../routes.php';                  // ③ 路線図を読み込む（$router に登録される）
$response = $router->dispatch($request);  // ④ 交換手に丸投げ → 返事をもらう
$response->send();                         // 最後に返事を送る
```

`Request::fromGlobals()` は、PHPがもともと持っている `$_GET` `$_POST` `$_SERVER` などの
**バラバラな情報を1つの箱（Request）にまとめる**作業です。以降アプリは、この箱越しにだけ入力を触ります（生の `$_GET` を直接触らない）。

### ステップ③ 交換手：Router がルートを探す（resolve）

[Router::resolve()](../../src/Presentation/Routing/Router.php#L59) が「どのルートか」を探します。探す順番が大事です。

```
1) 静的ルートを先に見る（一瞬で判定できる）
   → /thread/5 にピッタリ同じ固定URLはある？ → ない

2) 動的ルートを上から順に試す
   → /thread/{id} の形に当てはまる？ → 当てはまる！
   → {id} の部分から 5 を抜き出す → params = ['id' => '5']

   見つかったので RouteMatch（ルート＋抜き出した値）を返す
```

**なぜ静的が先か？** 静的URLは「完全一致するか？」をパッと調べられて速いからです（連想配列の一発引き）。
動的は1本ずつ正規表現で照合するので、その分コストがかかります。だから「速い方から試す」順番になっています。

#### 見つからないとき・メソッド違いのとき

- どのルートにも当てはまらない → **`NotFoundException`（404 Not Found）**
- URLは合うのに**メソッドだけ違う**（例：`/api/thread` を GET で叩いた。本当は POST 用）
  → **`MethodNotAllowedException`（405 Method Not Allowed）**
  このとき「本当はどのメソッドならOKか」を `Allow` ヘッダーで教えてあげます。

### ステップ④ {id} はどうやって抜き出す？（正規表現コンパイル）

`/thread/{id}` のような文字列は、そのままでは「`/thread/5` と一致するか」を判定できません。
そこで [Route.php](../../src/Presentation/Routing/Route.php#L32) が、ルート登録時にこれを**正規表現（パターン照合の道具）に変換**しておきます。

```
/thread/{id}
      ↓ コンパイル（変換）
#^/thread/(?P<id>[^/]+)$#
```

- `(?P<id>...)` = 「ここに当てはまった文字を **id という名前で取っておく**」という書き方（名前付きキャプチャ）
- `[^/]+` = 「`/` 以外が1文字以上」＝1セグメント分

`/thread/{id:\d+}` の場合は `\d+`（数字だけ）が使われるので、`/thread/abc` には一致しません。

> この「変換」はルート登録のときに1回やるだけ。毎回のアクセスでは、できあがった正規表現で照合するだけなので速いです。

### ステップ⑤ 交換手が担当につなぐ（dispatch）

ルートが見つかったら [Router::dispatch()](../../src/Presentation/Routing/Router.php#L99) が、いよいよ担当を呼びます。
ただし、その前に**ミドルウェア（検問）**を通します。

```
リクエスト
  → [csrf 検問] → ... → [ThreadController::show()] → Response
        通れば次へ                  本体の処理
```

ミドルウェアは「**通すか・止めるか**」を決める通過ゲートです（次の章で説明）。
検問を全部通り抜けると、最後に担当の処理（コントローラー）が呼ばれます。

```php
[$class, $action] = $match->route->handler;     // [ThreadController, 'show']
$controller = new $class();                       // 担当を用意
$result = $controller->$action($request);         // show($request) を実行
```

### ステップ⑥ 担当が返事を作る（Controller → Response）

[ThreadController::show()](../../src/Presentation/Controller/ThreadController.php#L13) はこれだけ。

```php
public function show(Request $request): Response
{
    $id = $request->param('id');              // 抜き出した id（=5）を取り出す
    return Response::json(['thread_id' => $id]);  // JSONで返事を作る
}
```

`$request->param('id')` で、ステップ③で抜き出した `id` を受け取れます。
返事は `Response` という箱に入れて返します（`Response::json(...)` や `Response::html(...)`）。

### ステップ⑦ 受付が返事を送る

`dispatch()` が返した `Response` を、index.php の最後で送ります。

```php
$response->send();   // ステータス・ヘッダー・本文をまとめてブラウザへ
```

これで `GET /thread/5` → `{"thread_id":"5"}` がブラウザに返ります。完了です。

## 5. ミドルウェア（検問所）のしくみ

ミドルウェアは、コントローラーに着く**前後にはさむ共通処理**です。
「**ログインしてる？**」「**この送信は本物？（CSRF）**」のようなチェックを、各コントローラーに書かずに一箇所でやれます。

### たとえ：空港の保安検査

```
搭乗口（コントローラー）に行く前に…
  手荷物検査 → パスポート確認 → 搭乗口
   （ミドルウェア）（ミドルウェア）
```

- 問題なければ「次へどうぞ」と通す（`$next($request)` を呼ぶ）
- 問題があればその場で止めて追い返す（自分で `Response` を返す）

### 実際のコード（CSRFの例）

[CsrfMiddleware](../../src/Presentation/Routing/Middleware/CsrfMiddleware.php) はこうなっています。

```php
public function handle(Request $request, callable $next): Response
{
    if ($request->method() === 'POST') {              // POSTのときだけ検査
        $sent    = (string) $request->input('_csrf', '');
        $session = (string) ($_SESSION['_csrf'] ?? '');
        if ($session === '' || !hash_equals($session, $sent)) {
            return Response::error(403, '不正なリクエストです'); // ← 止める
        }
    }
    return $next($request);   // ← 通す（次へ進む）
}
```

`$next` は「**次の処理（次の検問、最後はコントローラー）**」を指します。
`$next($request)` を呼べば先に進み、呼ばずに `Response` を返せばそこで打ち切りです。

### 複数のミドルウェアはどう重なる？（玉ねぎ構造）

ルートに `['a', 'b']` と2つ付いていると、こう包まれます（**外→内→外**）。

```
 a の前半
 └ b の前半
   └ コントローラー本体
 ┌ b の後半
 a の後半
```

リクエストは外側（a）から入って内側（本体）へ、レスポンスは内側から外側へ戻ります。
玉ねぎの皮のように何重にも包む形なので「**ミドルウェアパイプライン**」と呼ばれます。
これを作っているのが [dispatch() の array_reduce 部分](../../src/Presentation/Routing/Router.php#L113) です（登録の逆順に内側から包んでいく）。

ミドルウェアは名前（文字列 `'csrf'`）で指定し、[MiddlewareFactory](../../src/Presentation/Routing/MiddlewareFactory.php) が名前から実物を作ります。
`'rate_limit:post'` のように `:` で引数を渡す書き方にも対応しています。

## 6. エラーのときは？（404 / 405 / 500）

[index.php](../../public/index.php#L20) が、Router からの例外を**HTTPの返事に翻訳**します。

| 状況 | 投げられる例外 | ブラウザへの返事 |
|---|---|---|
| URLがどのルートにも無い | `NotFoundException` | **404** Not Found |
| URLは合うがメソッド違い | `MethodNotAllowedException` | **405** Method Not Allowed（`Allow` ヘッダー付き） |
| プログラム内で予期せぬエラー | その他すべての例外 | **500** Internal Server Error |

500のときは、開発中（`APP_ENV=development`）だけ詳しいエラー内容を表示し、
本番では `Internal Server Error` とだけ出して中身を隠します（情報漏れ防止）。
エラー文言は必ず `htmlspecialchars` でエスケープしてから出すので、入力をそのまま画面に出して攻撃される（XSS）こともありません。

## 7. まとめ（10秒で思い出す用）

1. **入口は1つ**：すべてのURLは nginx → `public/index.php` に集まる。
2. **路線図を読む**：`routes.php` が「URL → 担当」を登録する。
3. **交換手が探す**：`Router::resolve()` が静的→動的の順でルートを探し、`{id}` を抜き出す。
4. **検問を通す**：`Router::dispatch()` がミドルウェアを玉ねぎ状に通してから担当を呼ぶ。
5. **担当が返事を作る**：Controller が `Response` を作る。
6. **返事を送る**：`Response::send()` でブラウザへ。
7. **ダメなら翻訳**：見つからなければ 404、メソッド違いは 405、想定外は 500。
