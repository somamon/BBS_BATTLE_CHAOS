# 14. 自作ルーティング 実装設計

外部ルーティングライブラリを採用せず、ルーターを**自作**する方針に決定したことを受けた実装レベルの設計書。
要件・全体方針は [10_ルーティング・バリデーション設計.md](10_ルーティング・バリデーション設計.md) を親ドキュメントとし、本書はその具体的なクラス設計・アルゴリズム・移行手順を定義する。

## 0. 自作を選んだ理由（記録）

| 候補 | 不採用理由 |
|---|---|
| nikic/fast-route | stable(1.3.0)が2018年で停滞。2.0はbeta止まり |
| symfony/routing 8.x | PHP 8.4.1 必須（本番コンテナは PHP 8.3） |
| symfony/routing 7.x | 現役だが本要件に対して機能過多・学習コスト高 |
| league/route | FastRouteラッパー。更新頻度が低め |

> 本アプリのルート数は十数本規模で、必要機能は「静的＋単純な動的パラメータ＋メソッド振り分け＋ミドルウェア」のみ。
> フレームワーク級は過剰、停滞ライブラリは不安、という判断から **薄い自作ルーターが最適**と結論。外部依存を増やさない利点もある。

## 1. 現状実装とのギャップ

現状の [Router.php](../../src/Presentation/Routing/Router.php) は「メソッド＋パス完全一致」の最小実装。本設計が満たすべき差分は以下。

| 項目 | 現状 | 本設計 |
|---|---|---|
| 動的パラメータ `{id}` | ✕ 非対応 | ○ 正規表現変換で対応 |
| HTTPメソッド未許可 | 404扱い | **405** Method Not Allowed |
| ミドルウェア | ✕ | ○ ルート単位で適用 |
| Request/Response抽象 | ✕（生の `$_*`／`echo`） | ○ オブジェクトでラップ |
| コントローラ生成 | `new $controller()` 直書き | DIコンテナ経由（[13](13_導入ライブラリリファレンス.md) の php-di と接続可） |
| 例外時処理 | なし | 404/405/500 を明示処理 |

## 2. クラス構成

```
src/Presentation/Routing/
├── Router.php          ルート登録・解決・ディスパッチの中核
├── Route.php           1ルートの値オブジェクト（method, path, handler, middlewares, 正規表現）
├── RouteMatch.php      マッチ結果（Route ＋ 抽出したパラメータ）
├── RouterException.php → NotFoundException / MethodNotAllowedException
└── Middleware/
    └── MiddlewareInterface.php

src/Presentation/Http/
├── Request.php         $_GET/$_POST/$_SERVER をラップ
└── Response.php        ステータス・ヘッダ・ボディを保持し送信
```

> Request/Response は「ルーティング」より広い概念なので `Http/` に分離する。

## 3. Route（値オブジェクト）

1本のルートを表す不変オブジェクト。登録時にパスを正規表現へ**プリコンパイル**して保持する（毎リクエストの変換コストを避ける）。

| プロパティ | 型 | 説明 |
|---|---|---|
| `method` | string | `GET` / `POST` など（大文字正規化） |
| `path` | string | 元のパスパターン（例: `/thread/{id}`） |
| `handler` | array | `[ControllerClass::class, 'action']` |
| `middlewares` | string[] | 適用ミドルウェア名（例: `['csrf','rate_limit:post']`） |
| `regex` | string | コンパイル済み正規表現 |
| `paramNames` | string[] | プレースホルダ名の並び（例: `['id']`） |
| `isStatic` | bool | 動的部分を含まない＝完全一致で判定可能か |

### 3.1 パス → 正規表現 変換仕様

| 入力 | 出力（regex） |
|---|---|
| `/thread` | `#^/thread$#` |
| `/thread/{id}` | `#^/thread/(?P<id>[^/]+)$#` |
| `/api/thread/{id}/posts` | `#^/api/thread/(?P<id>[^/]+)/posts$#` |
| `/thread/{id:\d+}` | `#^/thread/(?P<id>\d+)$#`（型制約付き・任意拡張） |

変換ロジック（擬似）:

```php
// {name} または {name:regex} を名前付きキャプチャに変換
$regex = preg_replace_callback(
    '#\{(\w+)(?::([^}]+))?\}#',
    function ($m) use (&$paramNames) {
        $paramNames[] = $m[1];
        $pattern = $m[2] ?? '[^/]+';      // 制約未指定なら「スラッシュ以外1文字以上」
        return '(?P<' . $m[1] . '>' . $pattern . ')';
    },
    preg_quote($path, '#')   // 先に固定部分をエスケープ → その後プレースホルダを展開
);
$regex = '#^' . $regex . '$#';
```

> 注意: `preg_quote` を先にかけると `{` `}` までエスケープされるため、実装ではプレースホルダ抽出と固定部エスケープの順序に留意（固定セグメントのみ quote する方式が安全）。

## 4. Router（中核）

| メソッド | シグネチャ | 説明 |
|---|---|---|
| 登録（汎用） | `add(string $method, string $path, array $handler, array $mw = []): void` | Route を生成して保持 |
| 登録（糖衣） | `get(...)` / `post(...)` | `add('GET'/'POST', ...)` |
| 解決 | `resolve(string $method, string $path): RouteMatch` | マッチ探索。失敗時は例外を投げる |
| 実行 | `dispatch(Request $request): Response` | resolve → ミドルウェア → ハンドラ実行 |

### 4.1 resolve のアルゴリズム

```
1. パス末尾の / を正規化（"/thread/" → "/thread"。ただしルート "/" は除く）
2. 静的ルート table[method][path] を O(1) で先に引く（高速パス）
3. 該当なければ動的ルートを順に preg_match
   - マッチ → RouteMatch(route, params) を返す
4. パスはどれかにマッチするが method が違う場合
   → 許可メソッドを集めて MethodNotAllowedException(allowed[]) を投げる（405用）
5. どのパスにもマッチしない
   → NotFoundException を投げる（404用）
```

**重要な分岐**: 「パスは一致するがメソッド不一致」を 404 ではなく **405**（`Allow` ヘッダ付き）にするため、メソッドで弾く前に「パス一致するルート集合」を一度走査する。

### 4.2 dispatch の流れ

```
resolve() で RouteMatch 取得
  ↓ params を Request にセット（$request->withParams($match->params)）
ミドルウェアを「外→内」に積んだクロージャパイプラインを構築
  ↓ 各ミドルウェアは next() を呼ぶか、自前で Response を返して短絡
最内でコントローラを実行
  ↓ コントローラ生成は DIコンテナ（無ければ new）
  ↓ アクションに Request を渡す
戻り値（Response or 文字列）を Response に正規化して返す
```

## 5. RouteMatch / 例外

### RouteMatch

| プロパティ | 説明 |
|---|---|
| `route` | マッチした Route |
| `params` | 抽出済みパスパラメータ（例: `['id' => '01HXYZ...']`） |

### 例外（→ index.php の集中ハンドラで HTTP ステータスに変換）

| 例外 | 発生条件 | HTTP |
|---|---|---|
| `NotFoundException` | どのルートにも一致しない | 404 |
| `MethodNotAllowedException` | パス一致・メソッド不一致（許可メソッド配列を保持） | 405（`Allow:` ヘッダ） |

## 6. Request / Response

[10_ルーティング・バリデーション設計.md](10_ルーティング・バリデーション設計.md) §1.6 の規約を実装に落とす。

### Request（抜粋）

| メソッド | 返却 | 実装の元 |
|---|---|---|
| `method()` | string | `$_SERVER['REQUEST_METHOD']`（`_method` で上書き可・任意） |
| `path()` | string | `parse_url(REQUEST_URI, PHP_URL_PATH)` |
| `query($k,$d=null)` | mixed | `$_GET` |
| `input($k,$d=null)` | mixed | `$_POST` |
| `param($k,$d=null)` | mixed | resolve で注入したパスパラメータ |
| `header($name)` | ?string | `$_SERVER['HTTP_*']` |
| `ip()` | string | `REMOTE_ADDR`（信頼時のみ `X-Forwarded-For`） |
| `isHtmx()` | bool | `HX-Request` ヘッダ有無 |

`fromGlobals(): self` ファクトリでスーパーグローバルから生成し、以降アプリ内では Request のみを使う（`$_GET` 等を直接触らない）。

### Response（抜粋）

| メソッド | 説明 |
|---|---|
| `html(string $body, int $status = 200)` | Content-Type: text/html で生成 |
| `redirect(string $url, int $status = 302)` | Location ヘッダ |
| `error(int $status, string $message)` | エラーHTML |
| `send(): void` | status・header・body を実際に出力（最後に1回だけ呼ぶ） |

## 7. ミドルウェア

```php
interface MiddlewareInterface
{
    // 通過させるなら $next($request) を返す。短絡するなら自前の Response を返す。
    public function handle(Request $request, callable $next): Response;
}
```

| 名前 | クラス | 役割 | 適用 |
|---|---|---|---|
| `csrf` | CsrfMiddleware | トークン検証、失敗で 403 | POST系 |
| `rate_limit:post` | RateLimitMiddleware | 投稿の連投制限 | POST /api/post |
| `rate_limit:thread` | RateLimitMiddleware | スレッド作成制限 | POST /api/thread |

パイプラインは `array_reduce` でクロージャを内側から外側へ合成して構築する。

## 8. ルート登録（routes.php の将来形）

現状（[routes.php](../../src/Presentation/Routing/routes.php)）はハンドラ2本のみ。将来形のイメージ:

```php
/** @var Router $router */
$router->get('/',                 [TopController::class, 'index']);
$router->get('/thread/{id}',      [ThreadController::class, 'show']);
$router->get('/thread/create',    [ThreadController::class, 'createForm']);
$router->post('/api/thread',      [ThreadController::class, 'create'], ['csrf', 'rate_limit:thread']);
$router->post('/api/post',        [PostController::class, 'create'],   ['csrf', 'rate_limit:post']);
$router->get('/api/thread/{id}/posts', [ThreadController::class, 'posts']);
```

> **静的ルートと動的ルートの登録順に注意**: `/thread/create` と `/thread/{id}` が両方ある場合、静的（`/thread/create`）を先に判定する必要がある。§4.1 の「静的を table で先引き」方式ならこの順序問題は自動的に回避される（静的が常に優先）。

## 9. index.php への組み込み

現状の [index.php](../../public/index.php) は `dispatch($method, $path)` を直接呼ぶだけ。本設計適用後の骨格（[10](10_ルーティング・バリデーション設計.md) §3 と整合）:

```php
declare(strict_types=1);
require __DIR__ . '/../vendor/autoload.php';

session_start();
$request = App\Presentation\Http\Request::fromGlobals();

$router = new App\Presentation\Routing\Router(/* DIコンテナ */);
require __DIR__ . '/../src/Presentation/Routing/routes.php';

try {
    $response = $router->dispatch($request);
} catch (NotFoundException) {
    $response = Response::error(404, 'Not Found');
} catch (MethodNotAllowedException $e) {
    $response = Response::error(405, 'Method Not Allowed')
        ->withHeader('Allow', implode(', ', $e->allowed));
} catch (Throwable $e) {
    // 本番は汎用500、開発は詳細（config/app.php の env で切替）
    $response = Response::error(500, 'Internal Server Error');
}

$response->send();
```

## 10. 段階的移行手順

既存の `/thread`・`/playground` を壊さずに移行する想定。

1. `Request` / `Response` を追加（既存 Router はまだ触らない）
2. `Route` / `RouteMatch` / 例外クラスを追加
3. `Router` を拡張：`add/get/post` のシグネチャに `$mw=[]` を追加、`resolve()` を新設、`dispatch(Request)` を新設（旧 `dispatch($method,$path)` は一時的に併存→撤去）
4. コントローラのアクションを `index(Request $request): Response` 形へ順次変更（[PlaygroundController](../../src/Presentation/Controller/PlaygroundController.php) / [ThreadController](../../src/Presentation/Controller/ThreadController.php)）
5. `index.php` を §9 の骨格に差し替え
6. ミドルウェア（csrf / rate_limit）を最後に追加

## 11. テスト観点（[09_開発フェーズ.md](09_開発フェーズ.md) のユニットテストに追加）

| 対象 | ケース |
|---|---|
| パス→regex変換 | 静的・単一パラメータ・複数パラメータ・型制約付き |
