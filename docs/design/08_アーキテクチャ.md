## 1. 採用方針：簡易MVC

このゲームは「掲示板 + 投資経済」というドメインだが、規模は1人〜少人数の MVP であり、レイヤを厚く積む（Domain / Application / Infrastructure を分離する）クリーンアーキテクチャは**過度**と判断する。代わりに、素の PHP で見通しよく動く**簡易MVC**を採用する。

レイヤは3つに絞る。

| レイヤ | 役割 | 名前空間 |
|---|---|---|
| **Presentation** | HTTP 入出力・ルーティング・CSRF・画面描画。Controller は薄く、View は素の PHP テンプレート | `App\Presentation\*` |
| **Model** | エンティティ＝データアクセス＝ゲームロジックを兼ねる層。減衰計算・投資処理など「掲示板が朽ち、株が動く」ルールはここに集約 | `App\Model\*` |
| **Config** | ゲーム定数（初期所持金・配分比率・減衰レート・フェーズ倍率・変異閾値など）の一元管理 | `App\Config\*` |

> **将来の拡張余地**: Model が肥大化したら、エンティティ（純粋なゲームロジック）と Repository（データアクセス）を分離し、さらに UseCase を切り出して `App\Domain` / `App\Application` へ段階的に育てられる。今は `InvestService` がその「UseCase 層の芽」にあたる。最初から層を割らないことで、MVP の実装速度を優先する。

ゲーム定数の `Config\Game` を追加した姿。

## 3. 主要コンポーネントの責務

### Presentation

| コンポーネント | 責務 |
|---|---|
| `public/index.php` | 唯一の入口。`session_start()` → `Request::fromGlobals()` → `Router` 生成・ルート読込 → `dispatch()` → `Response::send()`。404/405/500 を例外で振り分け |
| `Routing\Router` | パス＋メソッドからルート解決。静的ルートは O(1)、動的（`{id}` 等）は登録順に正規表現マッチ。ミドルウェアを外→内に合成してコントローラを呼ぶ |
| `Http\Request` | `$_GET/$_POST/$_SERVER` とルートパラメータのラッパ。`method()` / `path()` / `input()` / `param()` を提供 |
| `Http\Response` | ステータス・ヘッダ・本文の組み立て。`html()` / `redirect()` / `error()` ファクトリ |
| `Routing\Middleware\CsrfMiddleware` | POST 系のみ `$_SESSION['_csrf']` と入力 `_csrf` を `hash_equals` で照合。不一致は 403 |
| `Http\Csrf` | セッションにトークンを保持し、フォーム用 hidden input を生成 |
| **Controller**（薄い） | 入力の取り出し → Model / Service 呼び出し → View 描画 or リダイレクト、だけを行う。**ゲームロジックは置かない**。例: `InvestController` は残高・ログインを軽く確認して `InvestService::invest()` を呼び、結果でリダイレクト |
| **View** | 素の PHP テンプレート。`View::render($name, $data)` が `ob_start` でレンダリング、`View::e()` で全出力を `htmlspecialchars` エスケープ。`layout.php` に共通枠（ヘッダーに「いまの相場」と所持金）、各画面を `$content` で差し込む。Plates 導入可だが当面は自作レンダラで足りる |

### Model（エンティティ＋データアクセス＋ゲームロジック）

| Model | 責務 |
|---|---|
| `Database` | PDO 接続を1つだけ生成・共有（MySQL、`ERRMODE_EXCEPTION`） |
| `Ulid` | 主キー用 ULID の採番 |
| `Thread` | スレの取得・作成に加え、**遅延減衰**（`currentHp()` で読み取り時に現在HP算出、`decayAndPersist()` で stored_hp へ書き戻し・`last_decay_at=now`）、`status` 判定、変異閾値チェック、株評価額の素地（`total_shares`）を担う |
| `Post` | レスの取得・作成・遅延減衰。投資対象外なので株・変異は持たず、`POST_INIT_HP` から朽ちるのみ |
| `User` | 登録投資家。`email`（一意）・`password_hash`・`money`（初期 `INITIAL_MONEY`）。登録／ログイン照合のデータアクセス |
| `Holding` | `(user_id, thread_id)` の保有株数。配当配分と評価額算出のための持ち分 |
| `Investment` | 投資の監査ログ（`amount` と HP/配当/sink への内訳）。経済の追跡用、再計算には使わない |
| `WorldState` | 世界に1行の相場フェーズ。`phase` / `phase_multiplier` / `next_shift_at`。読み取り時に `now ≥ next_shift_at` なら遅延でフェーズ抽選 |
| `InvestService` | **投資ユースケースの単一トランザクション境界**（下記） |

### InvestService（投資の1トランザクション境界）

`invest(User $investor, Thread $thread, int $amount)` を**1つの DB トランザクション**で実行する。

1. 検証：ログイン済み・`amount ≥ MIN_INVEST`・残高以下・スレが `alive`。
2. 配分（[§2.2](20_リデザイン_朽ちると投資.md#22-投資株式モデルー経済の心臓)）：`SPLIT_HP 50%` を HP 回復、`SPLIT_DIVIDEND 30%` を**新株発行前の既存株主**へ持ち分比率で配当、`SPLIT_SINK 20%` を消滅。最初の投資者は既存株主が居ないので配当分も sink に合流。
3. HP 更新は `Thread::decayAndPersist()` で現在HPを確定してから `hp += 0.5×amount`（`max_hp` で頭打ち、超過は sink）。
4. 株発行：投資者へ `amount` 株（`holdings` 加算、`thread.total_shares += amount`）。
5. 変異判定（[§2.6](20_リデザイン_朽ちると投資.md#26-変異化け株ー-スレの成長イベント)）：`total_shares` が `MUTATION_TIERS` を超えたら `mutation_level++`・`max_hp` 引き上げ。**遅延でなくこのトランザクション内**で確定。
6. ログ：`investments` に内訳を記録。
7. いずれか失敗で**全ロールバック**。

### Config

| コンポーネント | 責務 |
|---|---|
| `Config\Game` | [§3 バランス定数](20_リデザイン_朽ちると投資.md#3-バランス定数appconfiggame-に集約)（`INITIAL_MONEY` / `SPLIT_*` / `THREAD_*` / `POST_*` / `PHASES` / `PHASE_*_SEC` / `MUTATION_*`）を定数で持つ単一の真実の源。チューニングはここ1か所で完結 |

## 4. 重要な設計判断

### ① 遅延減衰ロジックを Model に集約し二重化を防ぐ

cron で全行を減らさず「読むたびに経過分から現在HPを計算」する遅延減衰（[§2.1](20_リデザイン_朽ちると投資.md#21-遅延減衰lazy-decayー実装の肝)）は、一覧表示・詳細表示・投資処理・終局判定など**複数の入口から呼ばれる**。式が各所に散ると倍率や経過分の扱いがズレてバグる。

➡ 計算は `Thread::currentHp()` / `Post::currentHp()`（読み取り専用）と `decayAndPersist()`（確定・書き戻し）に**一本化**し、Controller / View / Service はこれを呼ぶだけにする。`phase_multiplier` を掛ける箇所もここに閉じる。

### ② 投資の整合性は単一トランザクション（InvestService）

投資1回で「出金・配当入金・HP更新・株発行・変異・ログ」と複数テーブルが連動する。途中失敗で配当だけ入って株が発行されない等が起きると経済が壊れる。

➡ これらを `InvestService::invest()` の**1トランザクション境界**にまとめ、失敗時は全ロールバック。Controller はトランザクションを意識しない。

### ③ ゲーム定数の一元管理

減衰レート・配分比率・初期HP・フェーズ倍率・変異閾値は**実プレイで何度も調整する**（[§9 残課題 #3](20_リデザイン_朽ちると投資.md#9-残課題)）。値が Model や View に散ると調整が事故になる。

➡ すべて `Config\Game` の定数に集約。Model も Service も View もここを参照する。

### ④ 世界フェーズ／変異の責務配置

- **世界フェーズ（相場天候）**は `WorldState` Model が担う。cron を使わず、アクセス時に `now ≥ next_shift_at` なら**遅延でフェーズを抽選**し `phase_multiplier` を更新（[§2.5](20_リデザイン_朽ちると投資.md#25-世界フェーズ相場天候ー市場全体のボラティリティ)）。減衰式はこの倍率を読むだけ。
- **変異（化け株）**は投資イベント駆動なので `WorldState` でなく `InvestService` のトランザクション内で `Thread` の `mutation_level` を上げる（[§2.6](20_リデザイン_朽ちると投資.md#26-変異化け株ー-スレの成長イベント)）。「相場環境＝世界の遅延」「スレの成長＝投資イベント」と責務を分ける。

## 5. 依存方向・原則

```
Presentation ──→ Model ──→ Config
 (Controller/View/Http/Routing)   (Thread/Post/User/Holding/
                                   Investment/WorldState/InvestService)
```

- **一方向のみ**: Presentation は Model を呼べるが、Model は Presentation を知らない。Model は Config を参照するが、Config はどこにも依存しない。
- **Controller にロジックを置かない**: 入力取得・Model/Service 呼び出し・描画/リダイレクトのみ。減衰・配当・変異・抽選はすべて Model / Service 側。
- **ゲームルールの単一の源**: 減衰は `Thread/Post`、投資整合性は `InvestService`、定数は `Config\Game`。同じ計算を2か所に書かない。

## 6. リクエストライフサイクル

```
HTTP Request
  │
  ▼
public/index.php  ── session_start() / Request::fromGlobals()
  │                  Router 生成 + routes.php 読込
  ▼
Router::dispatch()
  │  resolve(method, path)  → 静的O(1) / 動的正規表現で Route 解決
  │                           （不一致は NotFoundException / MethodNotAllowedException）
  ▼
Middleware パイプライン（外→内に合成）
  │  CsrfMiddleware ── POST のみ _csrf を hash_equals 照合（不一致 403）
  ▼
Controller::action(Request)   ← 薄い：入力取得のみ
  │
  ▼
Model / InvestService          ← ゲームロジックの本体
  │  Thread::currentHp() / decayAndPersist()（遅延減衰・フェーズ倍率）
  │  WorldState（遅延フェーズ抽選）
  │  InvestService::invest()（1トランザクション：出金→配当→HP→株→変異→ログ）
  │  Database(PDO) 経由で永続化
  ▼
View::render()                 ← layout.php に $content を差し込み、e() でエスケープ
  │
  ▼
Response（html / redirect / error）
  │
  ▼
Response::send()  ── 404/405/500 は index.php の try/catch で整形
```

## 7. 補足

- **Composer / PSR-4**: `composer.json` で `App\ → src/` を1本マッピング（`Config\ → src/Config`、`Model\ → src/Model`、`Presentation\ → src/Presentation`）。
- **エラーハンドリング**: バリデーション失敗は Controller が 422、想定内のドメイン失敗（残高不足・dead 投資など）は Service が例外→Controller がメッセージ表示、未捕捉例外は `index.php` の `try/catch` で 500（開発時のみ詳細、本番は定型文）。
- **終局判定**（[§8](20_リデザイン_朽ちると投資.md#8-経済特性重要な留意点)）も cron 不要の遅延方式。投資／閲覧のたびに「全スレ dead」または「所持金合計 < `MIN_INVEST`」を軽くチェックする責務は Model 側に置く。
