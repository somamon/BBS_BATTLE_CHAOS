# 状態遷移・相場イベント設計

> 本書は [`20_リデザイン_朽ちると投資.md`](20_リデザイン_朽ちると投資.md) を正典とし、
> 「朽ちる掲示板と投資経済」の**動的な状態変化**を体系化する。
> cron を使わず、アクセス（読み書き）を起点に状態を遅延計算・遅延遷移するのが基本方針。
> 旧「カオス連鎖イベント」「汚染」「お題」は**将来拡張**（§7、正典 §10）として後付けする位置づけに格下げした。

---

## 1. 状態遷移の全体像

このゲームには「時間で進む状態」と「イベントで動く状態」が混在する。
cron を持たないため、すべての状態は**いずれかのアクセス時に必要なだけ計算・確定**される。

| 対象 | 状態 | 駆動方式 | 参照 |
|---|---|---|---|
| スレ／レスの HP | `alive` → `dead` | 遅延減衰（時間） | §2 |
| 世界フェーズ | `boom` / `calm` / `storm` / `crash` | 遅延遷移（時間・抽選） | §3 |
| スレの変異 | `mutation_level` 0 → 1 → 2 → 3 | イベント駆動（投資TX内） | §4 |
| ラウンド | 進行中 → 終局 | 遅延判定（アクセス時チェック） | §6 |

> 駆動方式は2系統だけ。**時間で進むもの（HP・フェーズ・終局）は「読むたびに計算」**、
> **不可逆に増えるもの（変異）は投資トランザクション内で確定**する。

## 2. 遅延減衰モデル（Lazy Decay）

スレもレスも個別の HP を持ち、時間で減り、0 で消滅（`dead`、復活不可）。
cron で全行を減らさず、**読むたびに経過時間から現在 HP を計算**する（正典 §2.1）。

### 2.1 現在 HP の計算式

```
現在HP = max(0, stored_hp − decay_per_min × phase_multiplier × 経過分数)
経過分数 = (now − last_decay_at) ÷ 60秒
phase_multiplier = 現在の世界フェーズの減衰倍率（§3。calm は 1.0）
```

- `stored_hp`（= `hp`カラム）/ `last_decay_at` は「HP を確定させた時刻のスナップショット」。
- レスは投資対象外なので変異・株を持たず、`POST_INIT_HP` から `POST_DECAY_PER_MIN` で減るのみ。
- スレは `THREAD_INIT_HP` から `THREAD_DECAY_PER_MIN` で減り、上限は `max_hp`（変異で上昇、§4）。
- 数値は正典 §3（バランス定数）に従う。

### 2.2 表示時と確定時の使い分け

| 局面 | 処理 | 書き込み |
|---|---|---|
| 一覧・詳細などの**表示** | `currentHp()` で計算するだけ | なし（純粋な読み取り） |
| 投資など**HP が変わるイベント** | `decayAndPersist()` で確定 | あり（スナップショット更新） |

`decayAndPersist()` の確定手順（HP変動イベントの直前に1回）:

```
1. 現在HP = currentHp()            // §2.1 で算出
2. stored_hp ← 現在HP             // 経過分を確定して書き戻し
3. last_decay_at ← now            // 新しいスナップショット時刻
4. （続けてイベント本体の HP 加算等を適用）
```

- 計算ロジックは Model に集約（`Thread::currentHp()` / `Thread::decayAndPersist()`）し二重化を防ぐ（正典 §7）。
- 表示は計算のみなので、閲覧だけでは DB を汚さない。

### 2.3 状態遷移 alive → dead

```
[alive] ──現在HP ≤ 0──▶ [dead]   （遷移は片方向・復活不可）
```

- 判定タイミング:
  - **表示時**: `currentHp() ≤ 0` のものは一覧・詳細から除外（その場で `dead` 書き込みは必須でない／除外表示で足りる）。
  - **イベント時**: `decayAndPersist()` の結果 `stored_hp = 0` なら `status='dead'` を確定。
- `dead` のスレには**投資不可・レス不可**（正典 §5）。`dead` のスレが保有する株は評価額 0（紙くず、正典 §2.3）。
- レスが `dead` でもスレ自体は生存しうる（独立 HP）。

> 設計判断: フェーズ倍率は「現在フェーズの倍率 × 経過分」で**近似**する（§3.3）。
> 厳密な区間積分はしない（MVP 許容、正典 §2.5）。

## 3. 世界フェーズ遷移（相場天候）

世界に1行だけの `world_state`（正典 §4）を持ち、**全スレの減衰に倍率を掛ける相場環境**を表す。

### 3.1 状態機械

```
        ┌──────────────────────────────┐
        ▼                              │
     [calm ×1.0] ◀──▶ [boom ×0.7]      │ 各遷移は
        ▲   │                          │ next_shift_at 到達時に
        │   ▼                          │ 再抽選（§3.2）
     [storm ×1.8] ◀──▶ [crash ×2.5] ───┘
```

| フェーズ | `phase_multiplier` | 体感 |
|---|---|---|
| `boom`（活況） | ×0.7 | 朽ちにくい。強気相場 |
| `calm`（平穏） | ×1.0 | 通常 |
| `storm`（嵐） | ×1.8 | 急速に朽ちる。延命コスト増 |
| `crash`（暴落） | ×2.5 | 投げ売り誘発。投資妙味も最大 |

- フェーズはトップ／一覧ヘッダーに常時表示（"いまの相場"）。嵐・暴落は投資家の行動を一気に動かす。

### 3.2 遅延遷移（cron 不要）

cron を使わず、任意アクセス時に次の手順でフェーズを進める（正典 §2.5 / §5「相場遷移」）。

```
1. world_state を読む
2. now ≥ next_shift_at か？
   - No  → 何もしない（現フェーズ継続）
   - Yes → 次フェーズを抽選（§3.4）
           phase ← 抽選結果
           phase_multiplier ← PHASES[phase]
           next_shift_at ← now + rand(PHASE_MIN_SEC, PHASE_MAX_SEC)
           updated_at ← now を確定
```

- `PHASE_MIN_SEC` / `PHASE_MAX_SEC` = 180 / 900 秒（正典 §3）。1フェーズ＝数分〜十数分。
- 抽選は「アクセスした最初の1リクエスト」が行えばよい（競合は更新の最終勝ち／軽い排他で十分。MVP は許容）。

### 3.3 近似の注記（重要）

- フェーズ切替時に**全スレを書き戻さない**。各スレの現在 HP は「**そのスレを読んだ時点の現フェーズ倍率** × 経過分」で近似する（§2.1）。
- すなわち、経過区間の途中でフェーズが変わっても過去区間に遡って倍率を分けない。
- 前提: フェーズは頻繁には変わらない（数分〜十数分単位）ため誤差は小さい。**厳密化（区間ごとの倍率積分）は将来拡張**。

### 3.4 遷移確率の例（実装時チューニング枠）

> 数値はバランス定数化前のたたき台。正典 §9-3「数値バランスは実プレイ後に調整」に従い `App\Config\Game` 側で集中管理する想定。

抽選は「現フェーズからの遷移重み」で加重ランダムに行う一例:

| 現フェーズ ＼ 次 | boom | calm | storm | crash |
|---|---|---|---|---|
| boom | 20 | 50 | 25 | 5 |
| calm | 25 | 40 | 25 | 10 |
| storm | 10 | 35 | 35 | 20 |
| crash | 5 | 35 | 30 | 30 |

- 同フェーズの再抽選（連続継続）も許す。
- 隣接（calm を経由）を厚めにし、boom↔crash の急反転は起こりにくくしている。

## 4. 変異イベント（化け株）

スレへの累計投資（＝`total_shares`）が閾値を超えると**変異**し、`mutation_level` が上がる（正典 §2.6）。
これは**時間ではなくイベント駆動**で、投資のトランザクション内で確定する。

### 4.1 変異 Lv 表

| Lv | 必要 `total_shares`（`MUTATION_TIERS`） | `max_hp`（`MUTATION_MAX_HP`） | 配当ボーナス（`MUTATION_DIV_BONUS`） | 演出 |
|---|---|---|---|---|
| 0 | — | 1000 | ×1.0 | 通常 |
| 1 | 500 | 2000 | ×1.1 | 「変異種」バッジ |
| 2 | 2000 | 4000 | ×1.2 | タイトル発光 |
| 3 | 8000 | 8000 | ×1.3 | 「特異点」化 |

### 4.2 判定ロジック（投資TX内）

投資で `total_shares` を加算した**直後**に判定する。

```
（株発行で total_shares が増えた後）
while mutation_level < 3
      かつ total_shares ≥ MUTATION_TIERS[mutation_level]:
    mutation_level ← mutation_level + 1
    max_hp ← MUTATION_MAX_HP[mutation_level]
    // ThreadMutated 相当のログ／演出フラグ（任意）
```

- `while` ループで、1回の大口投資が複数段の変異を**一気に飛ばす**ケースも吸収する。
- `max_hp` 引き上げは即時。**現在 HP は据え置き**（上限が伸びるだけで、以後の投資でより高くまで延命できる）。
- **配当ボーナス**は「変異後の Lv」を、その回**以降**の投資の配当配分(30%)に掛ける（§5 の順序参照）。

### 4.3 効果

- `max_hp` 上昇で延命の上限が伸び、長く戦える優良株になる。
- 配当ボーナスで株主への分配が厚くなる（その分 sink が減る）。
- 投資家は「変異目前のスレ」を先読みして仕込み、化け株を当てる。

## 5. 投資イベントの副作用連鎖

投資（`POST /thread/{id}/invest`、要登録）は複数の副作用を**1トランザクション**で実行する。
ロジックは単一ユースケース `InvestService::invest()` に集約し、トランザクション境界をそこに置く（正典 §7）。

### 5.1 前提チェック（TX 外でも可、最終確認は TX 内）

- ログイン済みか（未登録は不可）。
- `amount ≥ MIN_INVEST` かつ `amount ≤ 投資者の money`（残高不足は不可）。
- 対象スレが `alive` か（`dead` は不可）。

### 5.2 トランザクション内の順序

```
BEGIN
 1. スレ行を確定読み（必要なら FOR UPDATE）
 2. decayAndPersist(スレ)           … §2.2。減衰を確定し、現在HPと dead を最新化
                                      （ここで dead なら ABORT/ロールバック）
 3. 出金        : investor.money -= amount
 4. 配分計算    : to_hp=50% / to_dividend=30% / to_sink=20%（正典 §2.2）
 5. HP回復      : hp += to_hp（max_hp 超過分は to_sink へ合流）
 6. 配当        : 「株発行する前の既存株主」に持ち分比率で分配
                  ・分配額 = to_dividend × (1 + 変異ボーナス相当) を株主へ
                    （現 mutation_level の MUTATION_DIV_BONUS を適用、§4.1）
                  ・既存株主が居ない（＝最初の投資）→ 配当は丸ごと sink へ
                  ・ボーナスで増やした分は sink から差し引く
 7. sink        : to_sink を経済から消す（記録のみ、誰の残高にも入れない）
 8. 株発行      : holdings[investor] += amount ; thread.total_shares += amount
 9. 変異判定    : §4.2（total_shares 増加後に mutation_level++ / max_hp 更新）
10. ログ        : investments に amount / to_hp / to_dividend / to_sink を記録
COMMIT（失敗時は全ロールバック）
─────────────────────────────────────────
（COMMIT 後・TX外）
11. 終局判定    : §6 を軽くチェック（遅延方式）
```

### 5.3 順序の根拠

- **配当は株発行の「前」の株主集合に配る**（投資者本人は対象外。正典 §2.2）。よって §6（配当）は §8（株発行）より**前**。
- **変異判定は株発行の「後」**（`total_shares` が増えてから閾値判定する）。よって §9 は §8 の後。
- **配当ボーナスはその回の `mutation_level`**（変異判定前の値）を使う。つまり「同じ投資で踏んだ新 Lv のボーナスは、次の投資から効く」。
- HP 確定（§2 の `decayAndPersist`）を**最初**に行うことで、減衰と回復が二重計上されない。
- 終局判定は副作用が確定した後（COMMIT 後）に行えば十分（§6、遅延）。

## 6. 終局イベント

レス報酬 0・配当は再分配のため、お金は循環から退場し続ける（デフレ）。
構造上**有限ラウンド**であり、終局がある（正典 §8）。

### 6.1 終局条件（どちらか）

```
(A) 全スレッドが dead         ：alive なスレが 0 件
(B) 全登録ユーザーの money 合計 < MIN_INVEST  ：誰も投資できない
```

### 6.2 遅延判定

- cron 不要。**投資／閲覧のたびに軽くチェック**する遅延方式（正典 §8 末尾）。
- 判定は安価に（`SELECT COUNT(*) ... WHERE status='alive'` / `SELECT SUM(money) ...` 程度）。
- (A) の「alive 件数」は、表示時に `currentHp() ≤ 0` を `dead` とみなす遅延減衰と整合させる（厳密件数が要るタイミングでは確定読みを行う）。

### 6.3 リザルト確定

```
[進行中] ──条件(A) or (B)──▶ [終局]
```

- ランキング（`money + Σ株評価額`、正典 §2.3 / §5）を集計して確定し、**勝者＝総資産トップ**を「リザルト画面」で表示。
- 終局後は**閲覧のみ**。新ラウンド開始（全データをアーカイブ → 初期化）まで投資・投稿は停止。

## 7. 将来拡張（相場ニュース層として後付け）

コアを汚さず「相場イベント層」として段階的に追加できる（正典 §10）。**今回はコア外**。
コア相乗要素として採用済みなのは **世界フェーズ（§3）** と **変異（§4）** の2つのみ。

| 拡張 | 旧要素 | 投資ゲームでの意味 | 後付け方法（コア非破壊） |
|---|---|---|---|
| **連鎖イベント**（暴落／高騰／災害） | events | 投稿・投資を起点に確率で相場サプライズ。イベントログ＝市場ニュース | フェーズ抽選や投資TX後フックに確率発火を足す。HP一時増減として既存の減衰モデルに乗る |
| **汚染（ジャンク化）** | corruption | dead 接近等で汚染が進み、減衰加速＆配当率低下。優良株／不良株の選別軸 | スレに `corruption` 列を足し、`decay_per_min` と配当率に係数として掛けるだけ |
| **お題（テーマ株）** | challenges | 定期的にお題が出て、該当スレは期間中、減衰緩和・配当ボーナス。物色セクター | お題テーブルと時間ベース・ローテーションを追加。該当スレにフェーズ同様の倍率を上乗せ |

> いずれも「現在 HP 計算式（§2.1）の倍率」「配当配分（§5）」「相場ニュースのログ」という既存の口に係数・行を足す形で接続でき、
> 状態機械（§3）と投資連鎖（§5）の骨格を変えずに導入できる。旧「カオス連鎖イベント」はこの層に属する将来拡張である。
