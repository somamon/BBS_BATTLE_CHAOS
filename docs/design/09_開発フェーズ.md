# 開発フェーズ

> コア設計の正は [20_リデザイン_朽ちると投資.md](20_リデザイン_朽ちると投資.md)。本書はそれを実装に落とす**ロードマップ**。
> 数値・名称・配分は doc20 と一致させる（バランス定数は `App\Config\Game` に集約 / doc20 §3）。
> アーキテクチャは簡易MVC（`App\Model` + `Presentation/Controller|View`、doc20 §7）を踏襲する。

---

## 全体方針

- **MVPを最短で動かす → 段階拡張**。まず「朽ちる匿名掲示板」を成立させ、そこへ投資経済・相場・終局を順に積む。
- 各フェーズは**動作確認とビルド通過**（オートロード解決・`php -l` 構文チェック・主要画面が開く）を完了条件に含める。
- 既存の簡易MVC（スレ/レスCRUD）は**新スキーマに合わせて作り直す**。流用できる骨格（Router / Request / Response / Csrf / View / Database）はそのまま活かす。
- 減衰計算は Model に集約（`Thread::currentHp()` / `decayAndPersist()`）して二重実装を防ぐ（doc20 §2.1 / §7）。
- 投資は単一ユースケース `InvestService::invest()` に集約し、そこにトランザクション境界を置く（doc20 §2.2 / §7）。
- フェーズ遷移・終局判定は **cron 不使用の遅延方式**（アクセス時チェック）で実装する（doc20 §2.5 / §8）。

### フェーズ依存（概観）

```
Phase 1（朽ちる匿名掲示板）   ← MVP。これ単体で「立てて・書いて・朽ちる」が動く
   │
   ↓
Phase 2（認証＋投資経済）     ← users/holdings/investments を載せ、配当・ランキングまで
   │
   ↓
Phase 3（相場メカニクス）     ← 世界フェーズ遅延遷移＋変異Lv0〜3
   │
   ↓
Phase 4（有限ラウンド終局）   ← 終局判定＋リザルト＋（任意）アーカイブ/初期化
```

各フェーズは前フェーズの「完了条件(Doneの定義)」を満たして初めて着手する。横断作業（テスト・バランス調整・スパム対策）は §横断 を参照。

## Phase 1: スキーマ刷新 ＋ 朽ちる匿名掲示板（MVP）

**ゴール**: 登録不要で「スレ立て → レス投稿 → 時間で HP が減り 0 で消える」が成立する。doc20 §2.1（遅延減衰）の動く実体。

### 1-1: スキーマ刷新（マイグレーション）

旧 `001_create_threads.sql` / `002_create_posts.sql` は破棄し、doc20 §4 の全テーブルを作り直す。

| マイグレーション | 内容 | doc20 |
|---|---|---|
| `users` | id(ULID) / email(UNIQUE) / name / password_hash / money(500) / created_at | §4 users |
| `threads` | hp / max_hp / decay_per_min / mutation_level / total_shares / last_decay_at / status / post_count ほか | §4 threads |
| `posts` | thread_id / author_hash / author_id / content / hp / decay_per_min / last_decay_at / status | §4 posts |
| `holdings` | (user_id, thread_id) PK / shares | §4 holdings |
| `investments` | 監査ログ（amount / to_hp / to_dividend / to_sink） | §4 investments |
| `world_state` | 1行固定（id=1, phase='calm', phase_multiplier=1.0, next_shift_at） | §4 world_state |

- 索引は doc20 §4 の指定どおり（`uniq_users_email` / `idx_threads_status_created` / `idx_posts_thread` / `idx_holdings_thread` / `idx_investments_*`）。
- `world_state` は初期1行を INSERT。Phase 1 では `phase_multiplier` を読むだけ（遷移は Phase 3）。
- `users` / `holdings` / `investments` はこの段階で**作成だけ**しておき、利用は Phase 2。

### 1-2: 定数・Model 基盤

| タスク | 詳細 | doc20 |
|---|---|---|
| `App\Config\Game` | §3 のバランス定数を全て定義（HP/減衰/配分/フェーズ/変異） | §3 |
| `Thread` Model | `currentHp()`（遅延算出）/ `decayAndPersist()`（書き戻し＋確定）/ `isAlive()` | §2.1 |
| `Post` Model | `currentHp()` / `decayAndPersist()` / `isAlive()` | §2.1 |
| 減衰式の単一化 | `現在HP = max(0, stored_hp − decay_per_min × phase_multiplier × 経過分)` を Model 内 1 箇所に | §2.1 |

- `currentHp()` は読み取り専用（書き込みなし）。HP が変わるイベント時のみ `decayAndPersist()` で `stored_hp` へ書き戻し、`last_decay_at=now`、`現在HP≤0` なら `status='dead'`。
- dead は一覧から除外・**復活不可**。

### 1-3: 匿名掲示板の作り直し（Controller / View）

既存 `ThreadController` / `ResController` を新スキーマに合わせて再実装。

| UC | 実装 | doc20 |
|---|---|---|
| スレ作成 | HP満タン・`total_shares=0` で作成（登録者なら `creator_id`、MVPは匿名） | §5 |
| レス投稿 | スレ alive 確認 → post 作成 → `post_count++` | §5 |
| 一覧 | 現在フェーズ倍率で `currentHp()` を遅延計算、dead 除外、HPバー・勢い表示 | §5 / §6 |
| 詳細 | レス一覧（各 `currentHp()`、dead 除外）＋（投資パネルは Phase 2 で空枠） | §6 |

- ルート: `GET/POST /threads`, `GET /thread/{id}`, `POST /thread/{id}/posts`（doc20 §6）。
- POST 系は既存 `Csrf` を必須化。

**Phase 1 完了条件（Done）**
- マイグレーションが通り、新6テーブルが存在する（`users/holdings/investments/world_state` は空でよい）。
- ブラウザでスレ作成 → レス投稿ができる。
- 時間経過（または `last_decay_at` を過去にずらした検証）で一覧の HP が減り、0 で dead になり一覧から消える。
- 全 PHP ファイルが構文エラーなし・オートロード解決済み（ビルド通過）。

## Phase 2: 認証 ＋ 投資経済（経済の心臓）

**ゴール**: 登録ユーザーが株を買い、既存株主に配当が回り、総資産ランキングが出る。doc20 §2.2〜§2.4 / §8。

### 2-1: 認証（email + password）

| タスク | 詳細 | doc20 |
|---|---|---|
| 登録 | email + name + password 受取 → `password_hash()` → users 作成（money=500）→ セッションに user_id | §5 ユーザー登録 |
| ログイン | email で引く → `password_verify()` → セッションに user_id | §5 ログイン |
| セッション/ヘッダー | 未登録は「登録して投資」、登録済みは所持金を常時表示 | §6 |
| 失敗処理 | メール重複・形式不正・パス不一致を弾く | §5 |

- ルート: `GET/POST /register`, ログイン（`/login` 相当）。POST は CSRF 必須。

### 2-2: 投資ユースケース `InvestService::invest()`

doc20 §2.2 を**1トランザクション**で実装（出金・配当入金・HP更新・株発行・監査ログ）。失敗時ロールバック。

| 配分 | 割合 | 行き先 |
|---|---|---|
| HP回復 | 50% | `thread.hp += 0.5×amount`（`max_hp` 上限、超過は sink へ） |
| 配当 | 30% | **新規発行前の既存株主**へ持ち分比率で分配（投資者本人は対象外） |
| sink | 20% | 経済から消滅 |

- 投資者は `amount` 株を受領 → `holdings` 加算、`thread.total_shares += amount`。
- **最初の投資者**は既存株主が居ないため、その回の配当(30%)は sink に合流。
- 前提チェック: 要登録 / 残高 ≥ amount / amount ≥ `MIN_INVEST` / スレ alive。dead には投資不可。
- HP 更新前に `decayAndPersist()` で現在HPを確定してから加算する（遅延減衰との整合）。
- `investments` に `amount/to_hp/to_dividend/to_sink` を記録（監査ログ）。
- ルート: `POST /thread/{id}/invest`（要登録, CSRF）。スレ詳細の投資パネルを有効化。

### 2-3: マイページ ／ ランキング

| UC | 実装 | doc20 |
|---|---|---|
| マイページ | 所持金・保有株・各評価額・含み損益。評価額 = `(my_shares ÷ total_shares) × 現在HP` | §2.3 / §5 /me |
| ランキング | 全ユーザーの `money + Σ株評価額` を集計し降順 | §2.4 / §5 ranking |

- ルート: `GET /me`（要登録）, `GET /ranking`（匿名可）。

**Phase 2 完了条件（Done）**
- 登録 → ログイン → セッション維持ができ、ヘッダーに所持金が出る。
- 投資で「所持金 −amount／HP +50%（上限考慮）／既存株主へ配当30%／sink20%／株発行／total_shares 増加／investments 記録」が**1トランザクションで整合**して反映される（残高不足・dead・amount<MIN は弾かれる）。
- 最初の投資者で配当が sink に合流することを確認。
- マイページで評価額・含み損益が、ランキングで総資産順位が表示される。
- ビルド通過。

## Phase 3: 相場メカニクス（世界フェーズ ＋ 変異）

**ゴール**: 市場全体のボラティリティ（世界フェーズ）と化け株（変異）でゲームに波を作る。doc20 §2.5 / §2.6。

### 3-1: 世界フェーズ（遅延遷移 ＋ 減衰倍率）

| タスク | 詳細 | doc20 |
|---|---|---|
| 遅延遷移 | アクセス時に `now ≥ next_shift_at` ならフェーズを抽選し直し、`phase/phase_multiplier/next_shift_at` 更新（cron不要） | §2.5 |
| 抽選 | `boom0.7 / calm1.0 / storm1.8 / crash2.5`、継続秒は `PHASE_MIN_SEC`〜`PHASE_MAX_SEC`(180〜900) | §2.5 / §3 |
| 減衰への反映 | `currentHp()` の `phase_multiplier` に現フェーズ倍率を渡す（近似：フェーズ切替時に全スレ書き戻しはしない） | §2.5 |
| 表示 | トップ／一覧ヘッダーに現在フェーズを常時表示（"いまの相場"） | §2.5 / §6 |

### 3-2: 変異（化け株, Lv0〜3）

| タスク | 詳細 | doc20 |
|---|---|---|
| 変異判定 | `InvestService` のトランザクション内で `total_shares` が `MUTATION_TIERS`(500/2000/8000) 到達なら `mutation_level++` | §2.6 |
| `max_hp` 引き上げ | Lv1/2/3 = 2000/4000/8000（`MUTATION_MAX_HP`） | §2.6 / §3 |
| 配当ボーナス | 変異スレへの新規投資は配当(30%)に `MUTATION_DIV_BONUS`(1.1/1.2/1.3) を掛けて株主に厚く分配、その分 sink を減らす | §2.6 |
| 演出 | Lv1「変異種」バッジ / Lv2 タイトル発光 / Lv3「特異点」化 | §2.6 |

**Phase 3 完了条件（Done）**
- アクセスを跨いで `next_shift_at` 経過時にフェーズが切り替わり、ヘッダー表示と HP の減り方（倍率）が連動する。
- 投資で `total_shares` が閾値を超えると `mutation_level` が上がり、`max_hp` 上限と配当ボーナスが反映され、演出が出る。
- 配当ボーナス分だけ sink が減ることを確認。
- ビルド通過。

## Phase 4: 有限ラウンド終局 ＋ リザルト

**ゴール**: お金は循環で減るデフレ構造（doc20 §8）なので、終局を判定し勝者を確定する。

### 4-1: 終局判定（遅延方式）

| タスク | 詳細 | doc20 |
|---|---|---|
| 判定 | 投資/閲覧のたびに軽くチェック：①全スレが `dead`、または ②全ユーザー所持金合計 < `MIN_INVEST` | §8 |
| 終局フラグ | 成立で「終局」状態に遷移（以降は投資不可・閲覧のみ） | §8 |

### 4-2: リザルト画面

| タスク | 詳細 | doc20 |
|---|---|---|
| ランキング確定 | 終局時点の総資産（`money + Σ株評価額`）で順位を確定 | §8 |
| リザルト表示 | 勝者（総資産トップ）を表示する専用画面 | §8 |

### 4-3:（任意）アーカイブ / 初期化

| タスク | 詳細 | doc20 |
|---|---|---|
| アーカイブ | 終局ラウンドのデータを退避 | §8 |
| 初期化 | 新ラウンド開始（全データ初期化、world_state を calm に戻す） | §8 |

**Phase 4 完了条件（Done）**
- 全スレ dead もしくは総所持金 < `MIN_INVEST` で終局に入り、新規投資が止まる。
- リザルト画面で総資産トップが表示される。
- （任意採用時）アーカイブ → 初期化で新ラウンドを開始できる。
- ビルド通過。

## 横断タスク（全フェーズで継続）

### テスト

正典 CLAUDE.md 短期計画（ユニット／インテグレーション）に対応。

| 種別 | 対象 | フェーズ |
|---|---|---|
| ユニット | `Thread/Post` エンティティ、遅延減衰（経過分→現在HP、0でdead） | Phase 1〜 |
| ユニット | 投資配分（50/30/20、上限超過の sink 合流、最初の投資者の配当→sink） | Phase 2〜 |
| ユニット | 変異閾値（`MUTATION_TIERS` 到達で Lv++ / max_hp / 配当ボーナス） | Phase 3〜 |
| ユニット | フェーズ倍率を掛けた減衰（boom/calm/storm/crash） | Phase 3〜 |
| E2E | 登録 → 投資 → 配当（既存株主に分配される）の一連 | Phase 2〜 |
| E2E | スレ作成 → レス → 朽ちて dead → 一覧から消える | Phase 1〜 |

- ユースケースのテストはモックリポジトリ（あるいはテスト用DB）を用い、トランザクション境界（`InvestService`）の整合を検証する。

### バランス調整

- 数値は全て `App\Config\Game`（doc20 §3）に集約済み。実プレイ後に減衰・初期HP・配分%・フェーズ倍率・変異閾値をここだけで調整する（doc20 §9 #3）。

### スパム対策（検討）

- スレ作成のクールダウン等。MVP は無対策、実装後に判断（doc20 §9 #4）。`author_hash`（IPハッシュ）を識別の足がかりにできる。
