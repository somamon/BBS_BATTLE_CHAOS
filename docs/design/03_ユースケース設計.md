# ユースケース設計

> 本書は [20_リデザイン_朽ちると投資.md](20_リデザイン_朽ちると投資.md) を正典とし、その §5 ユースケース表を詳細化する。
> 数値・名称は正典と完全一致させる。バランス定数は `App\Config\Game`（[20 §3](20_リデザイン_朽ちると投資.md)）に集約。

ゲームは2層構造。**匿名掲示板層**（誰でも・無料・無報酬でスレ立て／レス投稿、HPで朽ちる）と、**投資家層**（email+password 登録、所持金で株を売買し総資産を競う）。本書では各ユースケースを「アクター／事前条件／基本フロー／事後条件／例外」で記述する。**最重要は「投資」**（[20 §2.2](20_リデザイン_朽ちると投資.md)）。

## UseCase 一覧

| UC | アクター | 認証 | 経済影響 |
|---|---|---|---|
| UC-01 ユーザー登録 | 訪問者 | 不要→登録 | money=500 付与 |
| UC-02 ログイン／ログアウト | 訪問者／登録ユーザー | 不要／要 | なし |
| UC-03 スレッド作成 | 誰でも | 匿名可 | なし（無料） |
| UC-04 レス投稿 | 誰でも | 匿名可 | なし（無報酬） |
| **UC-05 投資** | **登録ユーザー** | **要登録** | **出金・配当・sink・HP化・株発行** |
| UC-06 相場フェーズ遷移 | システム（遅延） | — | 全スレ減衰倍率変更 |

## UC-01 ユーザー登録

**アクター**: 訪問者（未登録）

**事前条件**: なし。

**基本フロー**

```
1. email + name + password を受け取る
2. バリデーション（email 形式、name 1〜50字、password 長さ）
3. email 重複チェック（uniq_users_email）
4. password_hash(password) で password_hash を生成
5. users を作成: id=ULID, money = INITIAL_MONEY(=500), created_at=now
6. セッションに user_id を保存（登録＝ログイン状態）
```

**事後条件**: users が1行追加され、所持金 500 を持つ投資家として参加する。セッションに user_id が入る。

**例外**
- メール重複 → 「既に登録済み」エラー、作成しない。
- 形式不正（email／name／password）→ バリデーションエラー。

> 経済メモ: 蛇口は登録時の +500 のみ。総マネー上限＝`500 × 登録人数`で、これ以降お金は増えない（[20 §8](20_リデザイン_朽ちると投資.md)）。

## UC-02 ログイン／ログアウト

**アクター**: 訪問者（ログイン）／登録ユーザー（ログアウト）

### ログイン

**事前条件**: 登録済みユーザーが存在する。

**基本フロー**

```
1. email + password を受け取る
2. users を email で引く
3. password_verify(password, password_hash) で照合
4. 成功ならセッションに user_id を保存
```

**事後条件**: セッションに user_id が入り、投資・マイページが利用可能になる。

**例外**
- メール不在 → 認証失敗（どちらが原因かは秘匿）。
- パスワード不一致 → 認証失敗。

### ログアウト

**基本フロー**: セッションから user_id を破棄。

**事後条件**: 未登録状態に戻る（閲覧・匿名投稿は引き続き可能、投資は不可）。

## UC-03 スレッド作成

**アクター**: 誰でも（匿名可）

**事前条件**: なし。匿名でもよい。

**基本フロー**

```
1. title を受け取りバリデーション（タイトル必須）
2. Thread を生成:
   - id=ULID, title
   - hp = THREAD_INIT_HP(=300)（満タン起点）, max_hp = THREAD_MAX_HP(=1000)
   - decay_per_min = THREAD_DECAY_PER_MIN(=5)
   - mutation_level = 0, total_shares = 0
   - status='alive', post_count=0
   - last_decay_at = now, created_at/updated_at = now
   - creator_id = 登録ユーザーならその id、匿名なら NULL（表示用）
3. threads に保存
```

**事後条件**: HP満タン・total_shares=0・株主ゼロのスレが alive で立つ。誰でも投資・レス可能になる。

**例外**: タイトル空 → バリデーションエラー。

> スレ作成は無料・無報酬。お金は動かない。

## UC-04 レス投稿

**アクター**: 誰でも（匿名可）

**事前条件**: 対象スレが存在し alive（現在HP>0）であること。

**基本フロー**

```
1. thread_id + content を受け取る
2. スレを取得し、現在HP を遅延計算（UC-07 と同じ式）して alive か判定
   現在HP = max(0, stored_hp − decay_per_min × phase_multiplier × 経過分)
   → 0以下なら status='dead' とし投稿不可
3. content をバリデーション（本文必須）
4. author_hash = IP のハッシュ（匿名識別）
5. Post を生成:
   - id=ULID, thread_id, author_hash, content
   - hp = POST_INIT_HP(=100), decay_per_min = POST_DECAY_PER_MIN(=5)
   - last_decay_at=now, status='alive', created_at=now
   - author_id = 登録ユーザーならその id（表示用）、匿名なら NULL
6. posts に保存
7. thread.post_count++（勢いの目安）
```

**事後条件**: レスが alive で追加され、post_count が +1 される。**報酬は一切発生しない**（お金は動かない）。レスは投資対象外で株・変異を持たず、`POST_INIT_HP` から減衰するのみ（放置20分で消滅）。

**例外**: スレ dead → 投稿拒否。本文空 → バリデーションエラー。

> レスのHP回復手段は無い（投資はスレ単位）。勢いが続くスレでも、個々のレスは時間で朽ちて消える。

## UC-05 投資【最重要】

**アクター**: 登録ユーザーのみ

**事前条件**:
- ログイン済み（セッションに user_id）。
- 投資額 `amount` が `MIN_INVEST(=1)` 以上かつ所持金以下。
- 対象スレが alive。

**基本フロー（全工程を1トランザクション）**

```
1. ログイン確認（未登録なら拒否）
2. amount 検証: amount ≥ MIN_INVEST(=1) かつ amount ≤ user.money
3. スレを行ロックで取得し、現在HP を遅延計算 → stored_hp に書き戻し
   → hp=現在HP, last_decay_at=now で確定（dead なら投資不可）
4. 配分を計算（[20 §2.2]）:
   - to_hp       = floor(amount × SPLIT_HP(=0.50))
   - to_dividend = floor(amount × SPLIT_DIVIDEND(=0.30))   ※変異なら配当ボーナス対象
   - to_sink     = amount − to_hp − to_dividend            ※端数も sink へ
5. 出金: user.money −= amount
6. HP回復: thread.hp += to_hp。max_hp 超過分は sink へ回す（hp は max_hp で cap）
7. 配当(30%): 「新規株発行【前】の既存株主」へ shares 比率で分配
   - 各株主への配当 = floor(to_dividend × (株主shares ÷ total_shares) × 配当ボーナス)
   - 変異スレ(mutation_level≥1)は MUTATION_DIV_BONUS(1.1/1.2/1.3) を掛けて厚く配り、増分は sink を減らして賄う
   - 既存株主が居ない（最初の投資者）場合、配当 30% はまるごと sink に合流
   - 投資者本人は配当対象外
8. sink(20%＋端数＋max_hp超過＋未配当分)は経済から消滅（記録のみ）
9. 株発行: holdings(user, thread) += amount、thread.total_shares += amount
10. 変異判定（イベント駆動・[20 §2.6]）:
    total_shares が MUTATION_TIERS(500/2000/8000) を超えたら
    mutation_level++、max_hp を MUTATION_MAX_HP(2000/4000/8000) へ引き上げ
11. investments に監査ログ（amount, to_hp, to_dividend, to_sink）
12. コミット（いずれか失敗で全ロールバック）
```

**事後条件**:
- 対象スレのHPが回復し延命される（変異閾値到達なら max_hp 上昇）。
- 既存株主の所持金が配当で増える。
- 投資者は `amount` 株を取得し株主になる（以降の後続投資の配当を受け取れる）。
- total_shares が増加。sink 分だけ循環マネーが減る（デフレ進行）。

**数値例（amount = 100、calm、変異なし、既存株主あり）**
- HP回復: `to_hp = 100 × 0.50 = 50` → thread.hp += 50
- 配当: `to_dividend = 100 × 0.30 = 30` → 既存株主へ持ち分比率で分配（例: 単独既存株主なら +30）
- sink: `to_sink = 100 − 50 − 30 = 20` → 経済から消滅
- 株発行: 投資者は +100 株、thread.total_shares += 100

**数値例（最初の投資者の場合）**: 既存株主が居ないため配当 30（=30）も sink に合流 → sink 合計 50、HP +50、+100 株。

**例外**
- 未登録 → 拒否。
- 残高不足（amount > money）→ 拒否。
- amount < MIN_INVEST(=1) → 拒否。
- スレ dead → 拒否。
- トランザクション中の失敗 → 全ロールバック（出金・配当・HP・株発行いずれも無効化）。

> 株の評価額 = `(my_shares ÷ total_shares) × スレ現在HP`（[20 §2.3]）。HPは時間で溶けるため、伸びるうちに乗り朽ちる前に配当を回収する投機ゲームになる。

## UC-06 相場フェーズ遷移

**アクター**: システム（遅延・任意アクセス時。cron不要）

**事前条件**: world_state が1行存在する（初期 phase='calm'）。

**基本フロー**

```
1. 任意のアクセス時に world_state を読む
2. now ≥ next_shift_at なら:
   a. PHASES(boom0.7 / calm1.0 / storm1.8 / crash2.5) からフェーズを抽選
   b. phase, phase_multiplier を更新
   c. next_shift_at = now + rand(PHASE_MIN_SEC=180 〜 PHASE_MAX_SEC=900)
   d. updated_at=now で保存
3. now < next_shift_at なら現フェーズを維持
```

**事後条件**: 現フェーズの `phase_multiplier` が確定し、以降の全スレ・全レスの遅延減衰計算に掛かる。

**例外**: なし（抽選は常に成功）。

> 近似: フェーズ切替時に全スレを書き戻さず、現フェーズ倍率×経過分で計算する（[20 §2.5]）。boom(×0.7)は朽ちにくく、crash(×2.5)は急速に朽ちて投資妙味が最大。

## UC-07 スレ一覧／詳細閲覧

**アクター**: 誰でも（匿名可）

**事前条件**: なし。

**基本フロー**

```
1. world_state を取得（必要なら UC-06 の遷移チェック）
2. スレ集合（一覧）または単一スレ＋レス（詳細）を取得
3. 各行の現在HP を遅延計算（書き込み不要・表示のみ）:
   現在HP = max(0, stored_hp − decay_per_min × phase_multiplier × 経過分)
   経過分 = (now − last_decay_at) ÷ 60
4. 現在HP ≤ 0 のもの（dead）は一覧から除外
5. 詳細では time価総額・HPバー・勢い・変異バッジ・投資パネルを表示
   - スレ時価総額 = 現在HP（全株の評価額合計に等しい）
   - レスも同式で現在HPを計算し、dead レスは除外
```

**事後条件**: 生存中のスレ／レスのみ、現在の相場フェーズ倍率を反映した現在HPで表示される。DBは更新しない（純粋な計算）。

**例外**: 対象スレが存在しない／dead → 詳細は 404 相当（一覧へ）。

## UC-08 資産ランキング

**アクター**: 誰でも（匿名可）

**事前条件**: なし。

**基本フロー**

```
1. 全登録ユーザーを取得
2. 各ユーザーの総資産を算出:
   総資産 = money + Σ(保有スレごとの株評価額)
   株評価額 = (my_shares ÷ thread.total_shares) × スレ現在HP（遅延計算）
   ※ dead スレ（現在HP=0）の株は評価額0＝紙くず
3. 総資産で降順ソートし、name と総資産を表示
```

**事後条件**: 全ユーザーが総資産順に並ぶ。表示のみでDBは更新しない。

**例外**: なし（ユーザー0なら空表示）。

## UC-09 マイページ

**アクター**: 登録ユーザー

**事前条件**: ログイン済み。

**基本フロー**

```
1. ログイン確認
2. user.money（所持金）を表示
3. holdings を引き、保有スレごとに:
   - 保有株数 my_shares
   - 現在の株評価額 = (my_shares ÷ total_shares) × スレ現在HP（遅延計算）
   - 含み損益（評価額と取得コストの差。investments のログから取得額を参照）
4. 総資産 = money + Σ評価額 を併記（UC-08 と同じ式）
```

**事後条件**: 所持金・各保有株・評価額・含み損益・総資産が一覧表示される。

**例外**: 未登録 → 拒否（ログインへ誘導）。

> 評価額は相場フェーズとスレHPで刻々変わる。dead スレの保有株は評価額0。

## UC-10 終局判定とリザルト

**アクター**: システム（遅延・投資/閲覧のたびに軽くチェック。cron不要）

**事前条件**: ラウンドが進行中。

**基本フロー**

```
1. 投資・閲覧のたびに終局条件を軽くチェック（[20 §8]）:
   - 全スレッドが dead（現在HP=0）、または
   - 全登録ユーザーの money 合計 < MIN_INVEST(=1)（誰も投資できない）
2. いずれか成立で終局へ移行
3. ランキングを確定（UC-08 と同じ集計）し、勝者（総資産トップ）を決定
4. リザルト画面で勝者・最終ランキングを表示
5. 以降は閲覧のみ（新ラウンド開始までデータは固定／アーカイブ→初期化）
```

**事後条件**: ランキングが確定し勝者が表示される。新規投資は受け付けない。

**例外**: 終局条件未成立 → 通常進行を継続。

> デフレ構造（投資ごとに HP化50%＋sink20% の計70%が循環マネーから退場）により、お金が枯れると全スレが朽ち、構造上「有限ラウンド」で必ず終局へ向かう（[20 §8](20_リデザイン_朽ちると投資.md)）。

## UseCase 依存関係

```
UC-05 投資（経済の心臓・1トランザクション）
  ├── users（出金・配当入金）
  ├── threads（現在HP確定・HP回復・total_shares・変異）
  ├── holdings（株発行・既存株主の配当配分）
  ├── investments（監査ログ）
  └── world_state（現在フェーズ倍率を遅延減衰に使用）

UC-07 一覧/詳細・UC-08 ランキング・UC-09 マイページ
  └── 現在HP／株評価額の遅延計算（書き込みなし）を共有

UC-06 相場遷移・UC-10 終局判定
  └── 任意アクセス時に遅延チェック（cron不要）
```
