# 16. 責務マップ（どのファイルが何を担当するか）

「このファイルは何をする所か」を一覧で引けるようにした責務リファレンス。
レイヤー構成・依存ルールの根拠は [08_アーキテクチャ.md](08_アーキテクチャ.md)、ルーティング詳細は [14](14_自作ルーティング実装設計.md)/[15](15_自作ルーティング実装ガイド.md)。

---

## 0. 一行原則（迷ったらこれ）

| レイヤー | 一言で言うと | やってよいこと | やってはいけないこと |
|---|---|---|---|
| **Presentation** | 「HTTPの入口と出口」 | 入力受取・検証・整形・HTML出力 | ビジネスロジック、DB直アクセス |
| **UseCase（Application）** | 「処理の段取り」 | Domainとリポジトリを呼んで1機能を完遂 | HTTPの知識、SQL、ドメインルールの実装 |
| **Domain** | 「業務ルールそのもの」 | 不変条件・計算・判定 | DB・HTTP・他レイヤーへの依存（**完全自己完結**） |
| **Infrastructure** | 「外の世界との接続」 | DB・乱数・イベント配信などの実装 | UseCase/Presentation を参照する |

依存の向きは **外→内のみ**（Presentation → UseCase → Domain ← Infrastructure）。Domain は誰にも依存しない。

> ⚠️ 命名の注意：[08](08_アーキテクチャ.md) は `src/UseCase/`（`App\UseCase\`）表記だが、現状ファイルシステムは `src/Application/UseCase/` になっている。**実装時にどちらかへ統一が必要**（本書は実態に合わせ `Application/UseCase` を採用）。

## 1. エントリ／プレゼンテーション層

| ファイル | 責務 | 備考 |
|---|---|---|
| `public/index.php` | **フロントコントローラー**。全リクエストの唯一の入口。オートロード→セッション開始→Request生成→DI組立→Router実行→Response送信→例外を404/405/500に変換 | [15 §10](15_自作ルーティング実装ガイド.md) |
| `src/Presentation/Routing/Router.php` | ルートの**登録・解決(resolve)・実行(dispatch)**。静的優先マッチ、405/404判定、ミドルウェア合成 | 中核。[15 §7](15_自作ルーティング実装ガイド.md) |
| `src/Presentation/Routing/routes.php` | **ルート定義表**。`method + path → [Controller, action] + middlewares` を列挙するだけ | ロジックを書かない |
| `src/Presentation/Routing/Route.php` | 1ルートの値オブジェクト。**パス→正規表現コンパイル**を担当 | `{id}` 変換はここ |
| `src/Presentation/Routing/RouteMatch.php` | マッチ結果（Route＋抽出パラメータ）の運搬 | 単なる入れ物 |
| `src/Presentation/Routing/NotFoundException.php` | 404を表す例外 | index.php がcatch |
| `src/Presentation/Routing/MethodNotAllowedException.php` | 405を表す例外（許可メソッド配列を保持） | `Allow`ヘッダ用 |
| `src/Presentation/Http/Request.php` | `$_SERVER/$_GET/$_POST/php://input` を**ラップ**。method/path/query/input/param/header/ip/isHtmx を提供 | アプリ本体はここ越しにのみ入力を触る |
| `src/Presentation/Http/Response.php` | ステータス・ヘッダ・ボディを保持し `send()` で**一括出力** | html/json/redirect/error |
| `src/Presentation/Middleware/CsrfMiddleware.php` | **CSRFトークン検証**。失敗で403 | POST系に適用 |
| `src/Presentation/Middleware/RateLimitMiddleware.php` | **連投制限**。セッションの最終時刻で判定 | post/thread別 |
| `src/Presentation/Controller/*Controller.php` | **入力の取り出し→Validator→UseCase呼び出し→View描画/リダイレクト**の司令塔。ロジックは持たない | 下表参照 |
| `src/Presentation/View/layout.php` | 共通レイアウト（head/header/footer） | Plates不採用なら素のPHP |
| `src/Presentation/View/**/ *.php` | 各ページ／HTMX部分テンプレート。**出力エスケープ(`e()`)はここ** | partialはレイアウトなし |

### コントローラ別の担当

| Controller | 担当エンドポイント群 | 呼ぶ主なUseCase |
|---|---|---|
| `TopController` | `/` | ListThreads / GetWorldState |
| `ThreadController` | `/thread/{id}`、`/thread/create`、`/api/thread` | CreateThread / GetThread / CollapseThread / MutateThread |
| `PostController` | `/api/post` | CreatePost / RevivePost / CorruptPost |
| `WorldController` | `/api/world/state` | GetWorldState / ChangeWeather / UpdatePhase |
| `EventController` | `/api/events/recent` | GetRecentEvents |
| `ChallengeController` | `/api/challenge/current` | GetCurrentChallenge / RotateChallenge |
| `PlaygroundController` | `/playground`（開発用） | なし（HTML返すだけ） |

> 既存の `ResControoler.php` は**スペルミス**（正＝`ResController`）。未使用なので整理対象。

## 2. ユースケース層（Application）

`src/Application/UseCase/<機能>/<動詞>.php`

| 責務 | 内容 |
|---|---|
| **1機能 = 1クラス = 1 `execute()`** | 「スレッド作成」「投稿作成」など、ユーザー操作1つを完遂する段取り役 |
| Domain と Repository を**協調**させる | エンティティ生成→ドメインサービス呼出→リポジトリで保存→イベント発火 |
| トランザクション境界 | 「1ユースケース＝1まとまり」の単位 |
| **持たない**もの | HTTPの知識、SQL、ドメインルールそのもの |

| 機能ディレクトリ | クラス（責務） |
|---|---|
| `Thread/` | CreateThread / GetThread / ListThreads / CollapseThread / MutateThread |
| `Post/` | CreatePost / RevivePost / CorruptPost |
| `World/` | GetWorldState / ChangeWeather / TriggerWorldEvent / UpdatePhase |
| `Challenge/` | GetCurrentChallenge / RotateChallenge |
| `Event/` | GetRecentEvents |

## 3. ドメイン層（Domain）

**何にも依存しない。業務ルールの本体。**

| ディレクトリ | 責務 | 例 |
|---|---|---|
| `Entity/` | **同一性を持つ業務オブジェクト**。状態と振る舞い、不変条件を持つ | Thread / Post / WorldState / Challenge / GameEvent |
| `ValueObject/` | **値そのもの**。生成時に正当性を検証（不正値は作れない） | ThreadId / ThreadTitle / PostContent / HitPoint / Corruption / Weather … |
| `Enum/` | 取りうる**状態の列挙** | ThreadStatus / WorldPhase / EventType … |
| `Event/` | **ドメインイベント**（起きた事実の表現） | PostCreated / ThreadCollapsed / WorldPhaseChanged … |
| `Service/` | 単一エンティティに収まらない**ドメイン計算・判定** | EventJudgeService / ChallengeJudgeService / WorldPhaseService |
| `Repository/` | **永続化の抽象（インターフェースのみ）**。実装はInfrastructure | ThreadRepositoryInterface … |

> ポイント：`Repository/` に**インターフェースだけ**置くのが依存逆転の肝。Domain は「保存できる」という契約だけ知り、SQLite/MySQL の事情は知らない。

---

## 4. インフラ層（Infrastructure）

**外の世界の実装。Domainのインターフェースを実装する形で内側に奉仕する。**

| ディレクトリ/ファイル | 責務 |
|---|---|
| `Persistence/SQLite/*Repository.php` | リポジトリIFの**SQLite実装**（実SQL） |
| `Persistence/SQLite/SQLiteConnection.php` | PDO接続の生成・保持 |
| `Persistence/MySQL/` | 将来のMySQL実装（同じIFを実装） |
| `Random/RandomGeneratorInterface.php` + `PhpRandomGenerator.php` | 乱数の抽象と実装（テストで差替え可能にする） |
| `Event/EventDispatcher.php` | ドメインイベントを**リスナーへ配信** |
| `Event/Listener/*Listener.php` | イベントに反応する副作用（隕石・浸食・崩壊連鎖など） |

## 6. 1リクエストで誰が何をするか（縦の流れ）

```
HTTP Request
  │
  ▼ public/index.php          … 入口。Request生成・DI組立・例外→HTTPステータス変換
  ▼ Router.php (resolve)      … パス/メソッド照合、{id}抽出、404/405判定
  ▼ Middleware                … CSRF/RateLimit（通過 or 短絡）
  ▼ *Controller               … 入力取得→Validator→UseCase呼出→View選択
  ▼ Application/UseCase        … 段取り：Domain操作＋Repository保存＋Event発火
  ▼ Domain(Entity/VO/Service)  … 業務ルール・不変条件・計算（ここが本体）
  ▼ Infrastructure(Repository) … 実際のDB読み書き／イベント配信
  ▲ View (+ e() でエスケープ)  … HTML（全体 or HTMX片）生成
  ▲ Response.php (send)        … status + header + body 出力
HTTP Response
```

**役割の一言まとめ**：
- 入口/出口の交通整理 → `index.php` + `Router` + `Request`/`Response`
- 入力の門番 → `Middleware` + `Validator`
- 司令塔（自分では計算しない）→ `Controller`
- 段取り → `UseCase`
- 業務ルールの本体 → `Domain`
- 外部との接続 → `Infrastructure`
- 見た目 → `View`

## 7. 「どこに書く？」早見表

| やりたいこと | 置き場所 |
|---|---|
| 新しいURLを追加 | `routes.php` に1行 |
| URLパラメータの取り出し | `Controller` で `$request->param('id')` |
| 入力チェック（必須・長さ等） | `Validator`（Presentation） |
| 「投稿を作成する」処理全体 | `Application/UseCase/Post/CreatePost` |
| 「HPが0で崩壊」などの判定ルール | `Domain/Entity` または `Domain/Service` |
| 値の正当性（タイトル1〜100字等） | `Domain/ValueObject` のコンストラクタ |
| DBへの保存SQL | `Infrastructure/Persistence/SQLite/*Repository` |
| イベントに対する副作用 | `Infrastructure/Event/Listener/*` |
| 画面・HTML | `Presentation/View/*` |
| 時刻・乱数（テストで固定したい値） | `Shared/Clock`・`Infrastructure/Random` |
