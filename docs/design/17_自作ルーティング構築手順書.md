# 17. 自作ルーティング 構築手順書（どのファイルをどの順で作るか）

「どのファイルを・どの順番で・何を書いて作るか」を、上から順に実行すればルーティングが完成する**作業手順書**。
完成形のフルコードは [15_自作ルーティング実装ガイド.md](15_自作ルーティング実装ガイド.md)、責務の根拠は [16_責務マップ.md](16_責務マップ.md)、設計判断は [14](14_自作ルーティング実装設計.md)。本書は「順番・依存・各段の確認」に特化する。

---

## 0. 作る順番の全体像

**依存の少ない部品（葉）から作り、最後に index.php で束ねる**のが鉄則。途中段階でも既存の `/thread`・`/playground` を壊さないように進める。

```
STEP1  Http/Request.php           ← 誰にも依存しない
STEP2  Http/Response.php          ← 誰にも依存しない
STEP3  Routing/例外2つ            ← 誰にも依存しない
STEP4  Routing/Route.php          ← (依存なし。正規表現コンパイル)
STEP5  Routing/RouteMatch.php     ← Route
STEP6  Routing/Router.php         ← Route, RouteMatch, 例外, Request, Response
STEP7  Middleware/ + Factory      ← Request, Response
STEP8  routes.php を新API化       ← Router
STEP9  Controller を Request/Response 化
STEP10 index.php を差し替え        ← 全部
STEP11 curl で検証
```

```
依存方向（→は「使う」）
index.php ──→ Router ──→ Route ──→ (なし)
   │           │   └──→ RouteMatch ──→ Route
   │           ├──→ 例外
   │           └──→ Middleware ──→ Request, Response
   └──→ Request, Response（どこからも使われる末端）
```

## STEP 1–2. Http層（Request / Response）を作る

**目的**：`$_SERVER` 等の生データと出力処理を、ここだけに閉じ込める。最初に作る理由は「誰にも依存せず、全員から使われる土台」だから。

- 作るもの：[15 §3](15_自作ルーティング実装ガイド.md)（Request）／[15 §4](15_自作ルーティング実装ガイド.md)（Response）のコードをそのまま。
- 名前空間：`App\Presentation\Http`
- ポイント：
  - `Request::fromGlobals()` が `REQUEST_URI` → `parse_url` でパス抽出、JSONボディ対応まで吸収。
  - `Response` は溜めて最後に `send()`。複数回 echo しない。

**この段の確認**（単体で試す。まだルーターには繋がない）:

```php
// 一時的な確認スクリプトで
$req = App\Presentation\Http\Request::fromGlobals();
var_dump($req->method(), $req->path());
App\Presentation\Http\Response::json(['ok' => true])->send();
```

## STEP 3. 例外2つを作る

**目的**：404／405を「戻り値」ではなく「例外」で表し、index.php の1か所で集中変換できるようにする。

- 作るもの：[15 §5](15_自作ルーティング実装ガイド.md) の `NotFoundException` / `MethodNotAllowedException`。
- ポイント：`MethodNotAllowedException` は**許可メソッド配列**(`$allowed`)を持つ（`Allow:` ヘッダに使う）。

---

## STEP 4–5. Route / RouteMatch を作る

**目的**：1ルートを表現し、**パス文字列 → 正規表現**へコンパイルする部分を独立させる。ここがルーティングの心臓。

- 作るもの：[15 §6](15_自作ルーティング実装ガイド.md)。
- ポイント：
  - `Route` のコンストラクタで `compile()` を呼び、`{id}` → `(?P<id>[^/]+)`、`{id:\d+}` → `(?P<id>\d+)` に変換して `regex`/`paramNames`/`isStatic` を確定。
  - `RouteMatch` は「Route ＋ 抽出済みパラメータ」を運ぶだけの入れ物。

**この段の確認**（変換が正しいか単体で）:

```php
$r = new App\Presentation\Routing\Route('GET', '/thread/{id}', ['X','show']);
var_dump($r->regex);        // "#^/thread/(?P<id>[^/]+)$#"
var_dump($r->paramNames);   // ["id"]
preg_match($r->regex, '/thread/42', $m);
var_dump($m['id']);         // "42"
```

## STEP 6. Router 本体を作る（既存を拡張）

**目的**：登録(`add/get/post`)・解決(`resolve`)・実行(`dispatch`)。現状の [Router.php](../../src/Presentation/Routing/Router.php) を置き換える。

- 作るもの：[15 §7](15_自作ルーティング実装ガイド.md) の Router 全文。
- 変更の要点（現状との差分）：
  - 静的ルートは `static[method][path]`、動的ルートは `dynamic[]` に分けて保持。
  - `resolve()`：①静的を `isset` で先引き → ②動的を `preg_match` → ③パス一致・メソッド違いを集めて 405、無ければ 404。
  - `dispatch(Request)`：パラメータを Request に注入 → ミドルウェアを `array_reduce` で内→外に合成 → コントローラ実行 → 文字列なら `Response::html` に正規化。
  - コンストラクタの `$resolver`：null なら `new`、php-di なら `fn($c)=>$container->get($c)`。

> **互換性のための移行テク**：いきなり旧 `dispatch($method,$path)` を消すと index.php が動かなくなる。STEP10 まで旧メソッドを残し、新 `dispatch(Request)` を併設 → index.php 差し替え後に旧を削除、の順で安全に移る。

**この段の確認**:

```php
$router = new Router();
$router->get('/thread/{id}', [SomeController::class, 'show']);
$match = $router->resolve('GET', '/thread/42');
var_dump($match->params);   // ["id" => "42"]
// resolve('POST','/thread/42') が MethodNotAllowedException を投げることも確認
```

## STEP 7. Middleware と Factory を作る

**目的**：ルート単位で前処理（CSRF等）を差し込めるようにする。最初は CSRF だけでよい。

- 作るもの：[15 §8](15_自作ルーティング実装ガイド.md) の `MiddlewareInterface` / `CsrfMiddleware` / `MiddlewareFactory`。
- ポイント：
  - `handle(Request, callable $next)` は「通すなら `$next($req)`、止めるなら自前 Response」。
  - `MiddlewareFactory::make('rate_limit:post')` のように `":"` で引数を分離。
  - RateLimit は後回し可（`routes.php` でミドルウェアを付けなければ動く）。

---

## STEP 8. routes.php を新APIへ更新

**目的**：ルート定義表を新しい登録方法に合わせる。

- 変更：[15 §9](15_自作ルーティング実装ガイド.md) を参考に。**静的→動的の順**で並べると読みやすい（解決順は Router 側で静的優先なので順序自体は不問だが可読性のため）。
- 既存2本（`/thread`・`/playground`）は必ず残す。

```php
/** @var Router $router */
$router->get('/playground', [PlaygroundController::class, 'index']);
$router->get('/thread',     [ThreadController::class, 'index']);
$router->get('/thread/{id}',[ThreadController::class, 'show']);              // 追加例
$router->post('/api/thread',[ThreadController::class, 'create'], ['csrf']); // MW付き例
```

---

## STEP 9. コントローラを Request/Response 化

**目的**：アクションのシグネチャを `action(Request $request): Response` に統一。

- 変更対象：[ThreadController.php](../../src/Presentation/Controller/ThreadController.php) / [PlaygroundController.php](../../src/Presentation/Controller/PlaygroundController.php)。
- before → after の型：

```php
// before（現状）: echo で直接出力・引数なし
public function index(): void { echo json_encode([...]); }

// after: Request を受け Response を返す
public function index(Request $request): Response {
    return Response::json(['message' => 'スレッド一覧']);
}
public function show(Request $request): Response {
    return Response::json(['thread_id' => $request->param('id')]);
}
```

- ポイント：出力は `echo` をやめ **Response を return**（送信は index.php の `send()` に一本化）。

---

## STEP 10. index.php を差し替え

**目的**：全部を束ね、例外を 404/405/500 に変換する最終形にする。

- 変更：[15 §10](15_自作ルーティング実装ガイド.md) の index.php 全文。
- 流れ：`require autoload` → `session_start()` → `Request::fromGlobals()` → `new Router()` → `require routes.php` → `try { dispatch }` で例外を Response に変換 → `send()`。
- ここまで来たら STEP6 で残した**旧 `dispatch($method,$path)` を削除**してよい。

## STEP 11. 動作確認（curl）

Docker 起動中・`vendor/autoload.php` 生成済みが前提。

```bash
curl -i  http://localhost:18723/playground          # 200 HTML
curl -s  http://localhost:18723/thread/42            # {"thread_id":"42"}
curl -i -X POST http://localhost:18723/thread/42     # 405 + Allow: GET
curl -i  http://localhost:18723/nope                 # 404
```

| 確認 | 期待 | 対応する実装 |
|---|---|---|
| 静的ルート | `/playground` が200 | Router 静的先引き |
| パラメータ抽出 | `/thread/42` で id=42 | Route 正規表現 + resolve |
| 405 | POST禁止ルートでAllow付き | resolve の405分岐 |
| 404 | 未定義パス | resolve の404分岐 |
| 型制約 | `/thread/{id:\d+}` に `abc` で404 | compile の `\d+` |

## 12. つまずきポイント early-warning

| 症状 | 原因 | 対処 |
|---|---|---|
| 404ばかり返る | 末尾スラッシュ未正規化 | `normalize()` で `rtrim('/')` |
| `Headers already sent` | View等で先に echo している | 出力は Response に集約、`send()` だけが出力 |
| パラメータが取れない | `preg_match` の名前付きキャプチャ名ミス | `paramNames` と `(?P<name>)` を一致させる |
| 静的より動的が先に当たる | 自前で foreach 順マッチにした | 静的は `isset` で先引きする設計を守る |
| 新クラスが Class not found | フォルダ名/namespace不一致（PSR-4） | `App\Presentation\Http` ↔ `src/Presentation/Http/` を一致させる |
