# ルーティング・バリデーション設計

> 本書は [20_リデザイン_朽ちると投資.md](20_リデザイン_朽ちると投資.md)（正典、特に §5 ユースケース / §6 画面・ルート）に従う。
> ルーター本体の作り方・内部実装は [14](14_自作ルーティング実装設計.md)〜[19](19_ルーティングのしくみ_やさしい解説.md) に委ねる。ここでは
> **「どんなルートを定義し、各入力をどう検証するか」** の設計のみを扱う。

本書の対象は **2層構造の掲示板**（[20 §1](20_リデザイン_朽ちると投資.md)）。
匿名でスレ立て・レス投稿ができる掲示板層と、`email + password` で登録して所持金 `500`（[20 §3](20_リデザイン_朽ちると投資.md) `INITIAL_MONEY`）で
スレッドの株を売買する投資家層が重なる。POST 系の書き込みはすべて CSRF 必須、投資・マイページは要登録（`auth`）ガードを掛ける。

---

## 1. ルーティング設計

### 1.1 方針

- **フロントコントローラ**: 全リクエストを `public/index.php` で受け、ブートストラップ後に `Router::dispatch()` へ渡す。
- **自作ルーター**: 静的ルート（`/threads` 等）は `[$method][$path]` ハッシュで O(1) 解決、動的ルート（`/thread/{id}` 等）は正規表現でマッチ。**静的が常に動的より優先**される（実装は [14](14_自作ルーティング実装設計.md)）。
- **登録 API**: `$router->get(path, [Controller::class, 'action'], [mw...])` / `$router->post(...)`。ハンドラは `[クラス, アクション]` の配列、第3引数にミドルウェア名の配列。
- **パスパラメータ**: `{id}` で名前付きキャプチャ。制約を付けたいときは `{id:\d+}` のように `{name:正規表現}` 形式（`Route::compile()` が `(?P<name>...)` に変換）。スレ/ユーザー ID は ULID（[20 §4](20_リデザイン_朽ちると投資.md)）なので原則 `{id}`（無制約）でよい。
- **ミドルウェア**: ルート単位で適用し、外→内に合成して実行（`Router::dispatch()`）。本設計では `csrf` と `auth` の2種。
  - `csrf`: POST のみ `_csrf` を `hash_equals()` で照合、不一致は **403**（`CsrfMiddleware`、実装済み）。
  - `auth`: セッションに `user_id` が無ければ `/login` へリダイレクト（要登録ガード。`MiddlewareFactory` に `'auth' => new AuthMiddleware()` を追加して有効化、実装は [14](14_自作ルーティング実装設計.md)）。

### 1.2 リクエスト処理フロー

```
HTTP Request
  ↓
public/index.php（オートロード／セッション開始／Config／DB接続／DI 組立／ルート登録）
  ↓
Request::fromGlobals()  … $_GET/$_POST/$_SERVER をラップ
  ↓
Router::dispatch($request)
  ├ resolve(method, path)  … 静的→動的の順でマッチ、{id} 等を params に抽出
  │    ├ 不一致         → NotFoundException（404）
  │    └ パス一致/メソッド違い → MethodNotAllowedException（405）
  ├ Middleware パイプライン（外→内）
  │    ├ csrf … POST の _csrf 照合（NG: 403）
  │    └ auth … 要登録チェック（未登録: /login へ 302）
  └ Controller::action($request)  … Validator で検証 → UseCase/Service 委譲 → View or redirect
  ↓
HTTP Response
```

`now ≥ next_shift_at` での相場フェーズ抽選（[20 §2.5](20_リデザイン_朽ちると投資.md)）や終局判定（[20 §8](20_リデザイン_朽ちると投資.md)）は **cron を使わず**、一覧・詳細・投資など任意アクセスの処理内で遅延実行する。ルーティング層には専用ルートを設けない。

### 1.3 ルート定義表（[20 §6](20_リデザイン_朽ちると投資.md) 準拠）

| メソッド | パス | コントローラー | アクション | ミドルウェア | 認証 |
|---|---|---|---|---|---|
| GET | `/threads` | `ThreadController` | `index` | — | 匿名可 |
| GET | `/thread/{id}` | `ThreadController` | `show` | — | 匿名可 |
| POST | `/threads` | `ThreadController` | `create` | `csrf` | 匿名可 |
| POST | `/thread/{id}/posts` | `ResController` | `create` | `csrf` | 匿名可 |
| POST | `/thread/{id}/invest` | `InvestController` | `invest` | `csrf`, `auth` | 要登録 |
| GET | `/register` | `AuthController` | `registerForm` | — | — |
| POST | `/register` | `AuthController` | `register` | `csrf` | — |
| GET | `/login` | `AuthController` | `loginForm` | — | — |
| POST | `/login` | `AuthController` | `login` | `csrf` | — |
| POST | `/logout` | `AuthController` | `logout` | `csrf`, `auth` | 要登録 |
| GET | `/me` | `AuthController` | `me` | `auth` | 要登録 |
| GET | `/ranking` | `RankingController` | `ranking` | — | 匿名可 |
| GET | `/result` | `RankingController` | `result` | — | 匿名可 |

> [20 §6](20_リデザイン_朽ちると投資.md) の表は GET `/register` を `GET/POST` 一括で記している。本書では作成フォーム表示（GET）と登録実行（POST）に分け、ログイン（`/login`）・ログアウト（`/logout`）・終局リザルト（`/result`、[20 §8](20_リデザイン_朽ちると投資.md)）を明示ルート化した。
> ヘッダーは未登録なら「登録して投資」ボタン、登録済みなら所持金と現在フェーズ（"いまの相場"）を常時表示する（[20 §2.5 / §6](20_リデザイン_朽ちると投資.md)）。

#### ルート定義の断片（`src/Presentation/Routing/routes.php`）

```php
// 掲示板層（匿名可）
$router->get ('/threads',            [ThreadController::class, 'index']);
$router->get ('/thread/{id}',        [ThreadController::class, 'show']);
$router->post('/threads',            [ThreadController::class, 'create'], ['csrf']);
$router->post('/thread/{id}/posts',  [ResController::class,    'create'], ['csrf']);

// 投資家層（要登録）
$router->post('/thread/{id}/invest', [InvestController::class, 'invest'], ['csrf', 'auth']);

// 認証
$router->get ('/register', [AuthController::class, 'registerForm']);
$router->post('/register', [AuthController::class, 'register'], ['csrf']);
$router->get ('/login',    [AuthController::class, 'loginForm']);
$router->post('/login',    [AuthController::class, 'login'],    ['csrf']);
$router->post('/logout',   [AuthController::class, 'logout'],   ['csrf', 'auth']);
$router->get ('/me',       [AuthController::class, 'me'],       ['auth']);

// 集計・終局
$router->get ('/ranking',  [RankingController::class, 'ranking']);
$router->get ('/result',   [RankingController::class, 'result']);
```

### 1.4 コントローラ責務の割り当て

| コントローラ | 担当ルート | 委譲先ユースケース（[20 §5](20_リデザイン_朽ちると投資.md)） |
|---|---|---|
| `ThreadController` | 一覧 `index` / 詳細 `show` / 作成 `create` | スレ作成・一覧/詳細（遅延減衰で現在HP算出、dead 除外） |
| `ResController` | レス投稿 `create` | レス投稿（スレ alive 確認 → post 作成 → `post_count++`） |
| `InvestController` | 投資 `invest` | `InvestService::invest()` を **1トランザクション**で実行＋変異判定（[20 §2.2 / §2.6](20_リデザイン_朽ちると投資.md)） |
| `AuthController` | 登録 / ログイン / ログアウト / マイページ | ユーザー登録（`password_hash()`, money=500）・ログイン（`password_verify()`）・マイページ（所持金・保有株・含み損益） |
| `RankingController` | ランキング `ranking` / リザルト `result` | 全ユーザーの `money + Σ株評価額` を集計し降順（終局時は確定値を `result` で表示） |

> 投資の金額授受・配当分配・HP更新・株発行・ログのトランザクション境界は `InvestService::invest()` に置く（[20 §2.2 / §7](20_リデザイン_朽ちると投資.md)）。コントローラはリクエスト取得・検証・委譲・応答のみ。

### 1.5 認証（`auth`）ガードの考え方

- 投資（`POST /thread/{id}/invest`）・マイページ（`GET /me`）・ログアウト（`POST /logout`）は **要登録**。これらに `auth` ミドルウェアを付ける。
- `auth` はセッションの `user_id` を見るだけ。**未登録なら `/login` へ 302 リダイレクト**（投資パネルからの導線は「登録して投資」へ）。
- 認証は email + password。ログイン成功時にセッションへ `user_id` を保存し（[20 §5 / §7](20_リデザイン_朽ちると投資.md)）、以降の `auth` 判定に使う。
- 匿名掲示板行為（スレ作成・レス投稿）は **登録不要**なので `auth` を付けない。登録ユーザーが行う場合のみ `creator_id` / `author_id` を表示用に紐付ける（[20 §4](20_リデザイン_朽ちると投資.md)）。
- 二重防御として、要登録のユースケース側（投資 Service 等）でも未登録・残高不足を弾く（[20 §5](20_リデザイン_朽ちると投資.md) の失敗条件）。

## 2. バリデーション設計

### 2.1 方針

- **サーバ側で必ず検証**する。HTML 側の `required` 等はあくまで補助。
- 検証は **Controller（または専用 Validator）で実施**し、入力の必須・型・長さ・形式・範囲を見る。OK なら UseCase/Service へ、NG ならエラー応答。
- 値域・整合性（残高 ≥ amount、スレ alive 等）は **Service/Entity 側のドメイン不変条件**で二重に守る（[20 §5](20_リデザイン_朽ちると投資.md) 失敗条件、[20 §2.2](20_リデザイン_朽ちると投資.md) はトランザクション内で再確認）。
- 入力は前後の空白を**トリム**してから検証する。

### 2.2 入力バリデーションルール一覧

#### スレ作成（`POST /threads`）

| フィールド | ルール | 備考 |
|---|---|---|
| `title` | 必須 / 文字列 / 1〜**255**文字 | `threads.title VARCHAR(255)`（[20 §4](20_リデザイン_朽ちると投資.md)）に一致 |
| `_csrf` | 必須 / セッション値と一致 | 不一致は 403（`CsrfMiddleware`） |

#### レス投稿（`POST /thread/{id}/posts`）

| フィールド | ルール | 備考 |
|---|---|---|
| `id`（パス） | 必須 / ULID 形式 | 不正・該当なしは 404 |
| `content` | 必須 / 文字列 / 1〜**2000**文字程度 | `posts.content TEXT`（[20 §4](20_リデザイン_朽ちると投資.md)）。長文上限はチューニング項目 |
| `_csrf` | 必須 / 一致 | — |

> 対象スレが `dead`（現在HP ≤ 0）の場合はレス投稿不可（[20 §5](20_リデザイン_朽ちると投資.md)）。これは検証の後、Service で alive 確認して弾く。

#### 投資（`POST /thread/{id}/invest`、要登録）

| フィールド | ルール | 備考 |
|---|---|---|
| `id`（パス） | 必須 / ULID 形式 / alive なスレ | dead には投資不可（[20 §2.2](20_リデザイン_朽ちると投資.md)） |
| `amount` | 必須 / 整数 / `≥ MIN_INVEST(=1)` / **所持金以下** | `MIN_INVEST` は [20 §3](20_リデザイン_朽ちると投資.md)。残高チェックはトランザクション内でも再確認 |
| `_csrf` | 必須 / 一致 | — |
| （認証） | セッション `user_id` 必須 | `auth` ミドルウェアで担保、未登録は `/login` |

#### ユーザー登録（`POST /register`）

| フィールド | ルール | 備考 |
|---|---|---|
| `email` | 必須 / メール形式 / **一意** | `users.email UNIQUE`（[20 §4](20_リデザイン_朽ちると投資.md)）。重複は登録失敗（[20 §5](20_リデザイン_朽ちると投資.md)） |
| `name` | 必須 / 文字列 / 1〜**50**文字 | `users.name VARCHAR(50)`、ランキング表示用 |
| `password` | 必須 / 最小長（例: 8文字以上） | `password_hash()` で保存（[20 §5 / §7](20_リデザイン_朽ちると投資.md)） |
| `_csrf` | 必須 / 一致 | — |

> `email` の一意性は形式チェック後に DB で確認する（DB の UNIQUE 制約も最終防衛線）。一意制約違反時は「このメールは登録済み」を 422 で再表示。

#### ログイン（`POST /login`）

| フィールド | ルール | 備考 |
|---|---|---|
| `email` | 必須 / メール形式 | 存在しなければ認証失敗 |
| `password` | 必須 | `password_verify()` で照合（[20 §5](20_リデザイン_朽ちると投資.md)） |
| `_csrf` | 必須 / 一致 | — |

> 認証失敗（メール不在 / パス不一致）は **どちらか特定できない汎用メッセージ**（「メールアドレスまたはパスワードが違います」）で 422 再表示。

#### ID 検証（`GET /thread/{id}` 等）

| フィールド | ルール | 失敗時 |
|---|---|---|
| `id`（パス） | 必須 / ULID 形式 / 該当 alive スレ存在 | 404（不正形式・該当なし・dead） |

### 2.3 エラー時の応答

| ケース | ステータス | 応答 |
|---|---|---|
| 入力バリデーション NG（フォーム系） | **422** | 入力フォームをエラーメッセージ付きで再表示（入力値は保持） |
| CSRF 不一致 | **403** | `Response::error(403, ...)`（`CsrfMiddleware`、実装済み） |
| 不正 ID・該当なし・dead 参照 | **404** | 404 ページ（`NotFoundException`） |
| メソッド不一致 | **405** | `MethodNotAllowedException` |
| 残高不足・dead への投資など意味的失敗 | 422 | エラーメッセージ付きで投資パネル/詳細を再表示 |
| 想定外例外 | 500 | グローバルハンドラで汎用エラーページ＋ログ |

- フォーム再表示は POST-Redirect-GET ではなく、Controller がエラー付きで View を返す（422）。成功時は対象ページへ **302 リダイレクト**（例: スレ作成成功 → `/thread/{id}`、投資成功 → 詳細）。
- 入力エラーは画面上部または各項目近傍にまとめて表示。CSRF トークン（`_csrf`）は再表示時にも埋め直す。

### 2.4 XSS・エスケープ方針（[20](20_リデザイン_朽ちると投資.md) 準拠）

- **保存は生の値**。`title` / `content` などはエスケープせずそのまま DB に格納する。
- **出力時に必ずエスケープ**。View の `e()` ヘルパー（`htmlspecialchars`）を通して描画し、投稿に含まれる HTML タグは「文字列として」表示する。
- バリデーション＝入力の**拒否**、エスケープ＝出力の**無害化**、トリム＝前後空白除去、と役割を分離する。
- `name`（ランキング表示）・`title`（一覧/詳細）・`content`（レス）はいずれもユーザー入力なので、表示箇所すべてでエスケープを徹底する。
