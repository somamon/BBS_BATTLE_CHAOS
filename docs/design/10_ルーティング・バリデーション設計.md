# ルーティング・バリデーション設計

## 1. ルーティング設計

### 1.1 方針

- **フロントコントローラーパターン**: 全リクエストを `public/index.php` で受ける
- **配列ベースのルートテーブル**: 設定ファイルに `[method, path, controller, action]` を列挙
- **動的パラメータ対応**: `/thread/{id}` のような可変部分を抽出
- **ミドルウェア対応**: CSRF・RateLimit をルート単位で適用

### 1.2 リクエスト処理フロー

```
HTTP Request
  ↓
public/index.php
  ↓
[1] Bootstrap（オートロード、Config読込、DB接続生成、DI組立）
  ↓
[2] Router::resolve($method, $path)
   ├ 該当ルート探索
   ├ パスパラメータ抽出（{id}など）
   └ ルート定義（controller, action, middlewares）を返却
  ↓
[3] Middleware パイプライン実行
   ├ CsrfMiddleware（POSTのみ）
   ├ RateLimitMiddleware
   └ 各 Middleware は通過 or レスポンス返却
  ↓
[4] Controller::action($request) 実行
   ├ リクエスト値の取得
   ├ Validator にかける
   │  ├ NG → 422 + エラーHTML返却
   │  └ OK → UseCase へ渡す
   ├ UseCase::execute() 実行
   └ View レンダリング or リダイレクト
  ↓
HTTP Response
```

### 1.3 ルートテーブル定義

`config/routes.php` で配列定義。

| メソッド | パス | コントローラー | アクション | ミドルウェア |
|---|---|---|---|---|
| GET | `/` | TopController | index | — |
| GET | `/thread/create` | ThreadController | createForm | — |
| GET | `/thread/{id}` | ThreadController | show | — |
| POST | `/api/thread` | ThreadController | create | csrf, rate_limit:thread |
| POST | `/api/post` | PostController | create | csrf, rate_limit:post |
| GET | `/api/thread/{id}/state` | ThreadController | state | — |
| GET | `/api/thread/{id}/posts` | ThreadController | posts | — |
| GET | `/api/world/state` | WorldController | state | — |
| GET | `/api/events/recent` | EventController | recent | — |
| GET | `/api/challenge/current` | ChallengeController | current | — |

### 1.4 パスマッチング仕様

| パターン | マッチ条件 |
|---|---|
| `/` | 完全一致 |
| `/thread/create` | 完全一致 |
| `/thread/{id}` | `/thread/XXXX` 形式、`XXXX` を `id` として抽出 |
| `/api/thread/{id}/posts` | `/api/thread/XXXX/posts` 形式 |

**実装方針**

ルートパスを正規表現に変換:

```
/thread/{id}        → #^/thread/(?P<id>[^/]+)$#
/api/thread/{id}/posts → #^/api/thread/(?P<id>[^/]+)/posts$#
```

`preg_match()` でマッチ確認、名前付きキャプチャでパラメータ抽出。

### 1.5 Router クラス責務

| メソッド | 入力 | 出力 | 説明 |
|---|---|---|---|
| `register(string $method, string $path, callable\|array $handler, array $middlewares = [])` | — | void | ルート登録 |
| `resolve(string $method, string $path)` | HTTP method, path | `RouteMatch` or null | マッチング |
| `dispatch(Request $request)` | Request | Response | ミドルウェア＋ハンドラー実行 |

### 1.6 Request / Response の抽象化

PHPの `$_GET`, `$_POST`, `$_SERVER` 等を直接触らず、Request オブジェクトでラップ。

**Request クラス**

| メソッド | 返却 | 説明 |
|---|---|---|
| `method()` | string | GET/POST等 |
| `path()` | string | URLパス |
| `query(string $key, $default = null)` | mixed | GETパラメータ |
| `input(string $key, $default = null)` | mixed | POSTパラメータ |
| `param(string $key, $default = null)` | mixed | URLパスパラメータ |
| `header(string $name)` | string\|null | HTTPヘッダー |
| `ip()` | string | クライアントIP |
| `isHtmx()` | bool | HX-Request ヘッダー有無 |

**Response クラス**

| メソッド | 説明 |
|---|---|
| `html(string $body, int $status = 200)` | HTML返却 |
| `redirect(string $url)` | 302リダイレクト |
| `error(int $status, string $message)` | エラーHTML返却 |

### 1.7 Controller の規約

| 規約 | 説明 |
|---|---|
| アクションの引数は `Request $request` のみ | パスパラメータは `$request->param('id')` |
| 戻り値は `Response` または HTML文字列 | 文字列はラッパーが自動で200ラップ |
| ビジネスロジックは書かない | 必ず UseCase に委譲 |
| バリデーションは Validator に委譲 | Controller では実行と分岐のみ |

## 2. バリデーション設計

### 2.1 方針

- **Validator クラス** を用意し、各リクエストに対応する Validator を定義
- **VO（Value Object）の生成時バリデーション** と二段構え
- **境界バリデーション**: Controller で形式チェック → Domain で意味的チェック

### 2.2 二段階バリデーション

| 段階 | 場所 | チェック内容 | 失敗時 |
|---|---|---|---|
| 入力バリデーション | Controller / Validator | 必須・型・長さ・形式 | 422 エラーHTML |
| ドメイン不変条件 | Entity / VO のコンストラクタ | 値域・整合性 | DomainException |

### 2.3 Validator クラス構造

**ベースクラス: Validator**

| メソッド | 説明 |
|---|---|
| `required(string $key)` | 必須チェック |
| `string(string $key)` | 文字列型チェック |
| `minLength(string $key, int $min)` | 最小長 |
| `maxLength(string $key, int $max)` | 最大長 |
| `integer(string $key)` | 整数チェック |
| `between(string $key, int $min, int $max)` | 範囲チェック |
| `ulid(string $key)` | ULID形式チェック |
| `inEnum(string $key, array $values)` | 列挙値チェック |
| `errors()` | エラー配列を返す |
| `isValid()` | エラーがないかどうか |

**具象 Validator 一覧**

| クラス | 対象エンドポイント | 検証項目 |
|---|---|---|
| CreateThreadValidator | POST /api/thread | title（必須、1〜100文字） |
| CreatePostValidator | POST /api/post | thread_id（必須、ULID）、content（必須、1〜1000文字） |
| ThreadIdValidator | GET /thread/{id} 等 | id（必須、ULID） |

### 2.4 バリデーションルール一覧

#### スレッド作成（POST /api/thread）

| フィールド | ルール | エラーメッセージ |
|---|---|---|
| title | required | タイトルを入力してください |
| title | string | タイトルは文字列で入力してください |
| title | minLength: 1 | タイトルを入力してください |
| title | maxLength: 100 | タイトルは100文字以内で入力してください |
| _csrf | required, match | 不正なリクエストです |

#### 投稿作成（POST /api/post）

| フィールド | ルール | エラーメッセージ |
|---|---|---|
| thread_id | required | スレッドIDが指定されていません |
| thread_id | ulid | スレッドIDの形式が不正です |
| content | required | 投稿内容を入力してください |
| content | string | 投稿内容は文字列で入力してください |
| content | minLength: 1 | 投稿内容を入力してください |
| content | maxLength: 1000 | 投稿は1000文字以内で入力してください |
| _csrf | required, match | 不正なリクエストです |

#### スレッド取得系（GET /thread/{id}, /api/thread/{id}/*）

| フィールド | ルール | エラーメッセージ |
|---|---|---|
| id | required | スレッドIDが指定されていません |
| id | ulid | スレッドIDの形式が不正です |

### 2.5 エラーレスポンス形式

#### HTMX 経由（POST /api/post 等）

HTTPステータス: 422

レスポンスHTML片:

```
<div class="form-errors">
  <ul>
    <li>投稿内容を入力してください</li>
    <li>投稿は1000文字以内で入力してください</li>
  </ul>
</div>
```

HTMX の `hx-target` で エラー表示エリアに差し込む。

#### 通常ページ（GET /thread/{id} で 不正ID）

HTTPステータス: 404
404ページHTML を返却。

### 2.6 サニタイズ vs バリデーション

| 種類 | 用途 | 実装 |
|---|---|---|
| バリデーション | 入力を「拒否する」 | Validator |
| エスケープ | 出力時に「無害化」 | View の `e()` ヘルパー（htmlspecialchars） |
| トリム | 前後空白除去 | Validator 内で実施 |

**重要原則**

- DB保存時は生の値を保存（エスケープしない）
- 表示時に必ずエスケープ
- 投稿内容に含まれるHTMLタグは「文字列として」保存・表示

### 2.7 CSRF 対策

| 項目 | 内容 |
|---|---|
| トークン生成 | `random_bytes(32)` → bin2hex → セッションに保存 |
| トークン検証 | `hash_equals()` でタイミング攻撃対策 |
| 検証対象 | POST/PUT/DELETE の全リクエスト |
| 検証失敗時 | 403 Forbidden |

### 2.8 レートリミット

| 対象 | 制限 | キー |
|---|---|---|
| POST /api/post | 10秒に1回 | `rate:post:{ip_hash}` |
| POST /api/thread | 60秒に1回 | `rate:thread:{ip_hash}` |

**実装方針（シンプル版）**

セッションに最終リクエスト時刻を記録 → 経過時間チェック。

```
$_SESSION['rate_limit']['post'] = timestamp
```

将来的にファイルベース or Redis に拡張可能。

### 2.9 IPアドレス取得

| ヘッダー | 用途 |
|---|---|
| REMOTE_ADDR | 基本のIP |
| HTTP_X_FORWARDED_FOR | プロキシ経由時（信頼できる場合のみ） |

**AuthorHash 生成**

```
hash('sha256', $ip . $salt)
```

`$salt` は config に固定値で持つ（同一IPに対して常に同じハッシュ）。

## 3. index.php の骨格設計

### 3.1 責務

| 順序 | 処理 |
|---|---|
| 1 | エラー設定（display_errors, error_reporting） |
| 2 | strict_types 宣言 |
| 3 | Composer オートロード読込 |
| 4 | セッション開始 |
| 5 | Config 読込 |
| 6 | DB接続生成 |
| 7 | 依存組み立て（Repository, UseCase, Controller） |
| 8 | Router にルート登録 |
| 9 | Request オブジェクト生成 |
| 10 | Router::dispatch($request) → Response |
| 11 | Response 送信 |
| 12 | 例外時はグローバルハンドラで500ページ |

### 3.2 グローバル例外処理

| 例外種別 | 処理 |
|---|---|
| ValidationException | Controller でcatch → 422 |
| NotFoundException | Router でcatch → 404 |
| DomainException | Controller でcatch → 適切なステータス |
| Throwable（その他） | グローバルハンドラ → 500 + ログ記録 |

### 3.3 開発・本番モード切替

| モード | display_errors | エラー画面 |
|---|---|---|
| development | On | 詳細表示 |
| production | Off | 汎用エラーページ |

`config/app.php` の `env` で切替。
